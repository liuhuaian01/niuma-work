## 一、研究概述

### 1.1 研究背景

超级牛马工作台当前处于v1.5阶段，太极引擎模块完成度约51%，25项功能基线完成度76%。产品覆盖9个导航Tab（首页/对话/广场/项目/记忆/文件/连接/实验室/办公室），目标服务5类用户。在推进v2.0之前，需要系统性验证：

- 产品核心价值主张是否击中用户真实痛点
- 不同用户群的差异化需求是否被充分覆盖
- 当前产品体验中的关键摩擦点在哪里

### 1.2 核心研究问题

| # | 研究问题 | 问题类型 | 预期产出 |
|:--:|----------|:--:|------|
| RQ1 | **各类用户当前使用什么工具完成日常AI工作？超级牛马替代了什么、补充了什么？** | 探索性 | 竞品替代地图 |
| RQ2 | **用户对"AI工作台"的心理模型是什么？超级牛马的9个Tab是否符合预期？** | 验证性 | 信息架构评估 |
| RQ3 | **首次用户体验(FTUE)中，哪些环节产生困惑或放弃？引导流程是否有效？** | 评估性 | FTUE问题清单 |
| RQ4 | **对话聊天模块的交互模式是否满足不同用户的工作流需求？** | 评估性 | 对话体验优化建议 |
| RQ5 | **记忆/意识模块（三层记忆系统）对用户是否可理解、有价值？** | 验证性 | 记忆模块改进方向 |
| RQ6 | **不同用户群对"广场"(Skills/Experts/Models)的需求优先级和期待是什么？** | 探索性 | 广场功能优先级矩阵 |

### 1.3 成功标准

研究本身成功的衡量指标：

| 指标 | 达标线 | 说明 |
|------|:---:|------|
| 招募完成率 | ≥90% | 每类用户至少实际参研3人 |
| 研究洞察采纳率 | ≥70% | 研究建议被产品/设计团队采纳并落地 |
| 关键假设验证覆盖 | 100% | RQ1-RQ6全部获得可行动结论 |
| 参与者多样性 | 5类全覆盖 | 不遗漏任何目标用户群 |

## 二、用户画像与招募

### 2.1 五类目标用户画像

#### 画像A：网文作者

| 维度 | 描述 |
|------|------|
| **典型角色** | 起点/七猫/番茄签约作者，日更4000-8000字 |
| **核心工作流** | 大纲→章节写作→修改→发布；同时管理多本书 |
| **当前工具链** | WPS/Word/石墨 + AI助手(Claude/DeepSeek) + 写作软件(墨者/大神码字) |
| **核心痛点** | 灵感碎片散落各处难以管理；角色/设定前后不一致；AI辅助写作时上下文断裂 |
| **对AI工作台的期待** | "一个能记住我所有角色和设定的AI写作搭档" |
| **技术素养** | 中等——会用AI工具但不懂技术参数 |

**招募条件：**
- 签约作者，至少完成1本≥30万字作品
- 目前使用至少1个AI工具辅助写作
- 每周写作≥4天

**计划招募：5人**

## 三、研究方法设计

采用**混合研究方法(Mixed Methods)**，分三个阶段执行：

### 3.1 研究阶段总览

```
阶段一：探索性研究（定性为主）
├── 方法1：半结构化深度访谈（n=23）
├── 方法2：日记研究（n=10，5天）
└── 产出：用户画像细化、JTBD地图、核心痛点假设

阶段二：验证性研究（定量为主）
├── 方法3：大规模问卷调查（n=150-200）
├── 方法4：竞品对比分析（5-8个竞品）
└── 产出：需求优先级矩阵、竞品定位图、假设验证报告

阶段三：评估性研究（行为观察）
├── 方法5：可用性测试（n=10，含眼动追踪）
├── 方法6：A/B原型测试（2-3个关键流程变体）
└── 产出：可用性问题清单、交互优化建议、A/B胜出方案
```

### 3.2 方法一：半结构化深度访谈

**访谈目标：** 理解用户的JTBD(Jobs-To-Be-Done)、现有工具链、对AI工作台的期待和心理模型。

**样本量：** 23人（画像A5 + B5 + C5 + D4 + E4）

**访谈大纲骨架：**

```
模块1：热身（5分钟）
- 自我介绍，日常一天是怎么度过的？
- 工作/创作中最大的"烦心事"是什么？

模块2：当前工具生态（15分钟）
- 请画出你日常使用的所有工具之间的关系图
- 哪两个工具之间切换最频繁？为什么？
- 如果有"魔法棒"可以消除一个重复操作，是什么？
- 你目前在哪些环节使用AI？效果怎么样？哪里不满意？

模块3：AI工作台认知（15分钟）
- 听到"AI工作台"这个词，你脑海中浮现什么画面？
- 如果有一个"超能力AI助手"，你最希望它能做什么？
- [展示超级牛马概念图] 你觉得这和你想象的一致吗？哪里有出入？

模块4：超级牛马体验反馈（20分钟）
- [展示9个Tab导航] 你觉得每个Tab大概是什么功能？
- 哪些Tab你第一眼就理解？哪些需要解释？
- 你最可能高频使用的3个Tab是？为什么？
- [展示记忆/意识模块] 这个概念你理解吗？对你有吸引力吗？

模块5：定价与推荐意愿（5分钟）
- 如果这个产品收费，你愿意付多少钱/月？
- 你会推荐给谁？为什么推荐/为什么不推荐？
```

### 3.3 方法二：日记研究

**目标：** 捕捉5天内用户与AI工具的日常交互模式，发现隐性需求。

**样本量：** 10人（每画像2人）

**日记模板：**

```
日期：[    ]
今日主要工作：[一句话描述]

1. 今天使用AI工具的次数：___次
2. 最满意的一次AI交互（截图+描述）：
3. 最不满意的一次AI交互（截图+描述）：
4. 今天有没有想做但AI工具做不到的事？
5. 今天在哪些工具之间切换了？切换时有没有觉得烦？
6. 是否有灵感/想法一闪而过但没来得及记录？
```

### 3.4 方法三：问卷调查

**目标：** 验证定性研究产生的假设，获取统计显著的需求优先级数据。

**样本量：** 150-200人（每画像≥30人）

**调查维度：**

| 维度 | 题目数 | 题型 | 示例题项 |
|------|:---:|------|------|
| 人口统计 | 5 | 单选 | 角色/行业/年限 |
| AI工具使用习惯 | 8 | 单选+多选 | 每日使用AI时长、常用功能 |
| 痛点强度 | 12 | Likert 5级 | "AI对话历史难以检索和复用" 1-5 |
| 功能期望 | 15 | Likert 5级 + MaxDiff | 9个Tab中选最重要和最不重要的 |
| 价格敏感度 | 3 | Van Westendorp | 太便宜/便宜/贵/太贵 四档 |
| NPS前置 | 2 | 0-10 + 开放题 | 推荐意愿+理由 |

**MaxDiff设计：** 9个Tab功能做MaxDiff实验（每次展示4个，选最重要和最不重要各1），每个受访者完成12组判断。用于无偏差地获取功能优先级。

### 3.5 方法四：竞品对比分析

**目标：** 理解超级牛马在竞争格局中的定位和差异化机会。

**竞品列表：**

| 竞品 | 类型 | 分析重点 |
|------|------|------|
| Coze (字节) | AI Bot构建平台 | 广场/技能市场的交互模式 |
| Cursor | AI编程工具 | 工作间/项目管理的设计模式 |
| Notion AI | AI增强文档 | 记忆/知识库的组织方式 |
| Monica | AI助手聚合 | 多模型切换的用户体验 |
| Dify | AI应用开发平台 | 工作流编排的可视化设计 |
| 通义千问 | 国产大模型 | 对话界面交互模式 |
| ChatGPT Desktop | AI桌面应用 | 桌面端体验标杆 |

**分析维度：**
- 信息架构对比（Tab/导航结构）
- FTUE引导流程对比（前5分钟体验）
- 记忆/历史管理机制对比
- 多工作间/多项目切换体验
- 定价模式对比

### 3.6 方法五：可用性测试

**目标：** 在原型上验证关键交互流程的可用性。

**样本量：** 10人（每画像2人）

**测试平台：** 基于 `niuma-neon-pulse-prototype.html` 构建可交互原型

**测试任务（6个核心任务）：**

| 任务 | 场景描述 | 成功标准 | 关键指标 |
|:--:|------|------|------|
| T1 | "你第一次打开超级牛马，请完成初始设置并发送第一条AI对话" | 成功发送消息 | 完成时间、求助次数、错误路径 |
| T2 | "你昨天和AI讨论过一个关于XX的话题，请把它找出来" | 定位到历史对话 | 搜索路径选择、完成时间 |
| T3 | "请创建一个新项目'XX'，并切换到这个项目工作" | 项目创建并切换成功 | 操作步骤数、是否使用工作间概念 |
| T4 | "你需要在广场找到一个适合写作的Expert，并使用它" | 找到并使用Expert | 搜索vs浏览、理解度 |
| T5 | "请开启/关闭一个Agent能力开关" | 找到设置并操作 | 路径效率、术语理解 |
| T6 | "查看你本月的Token使用情况" | 成功查看Token仪表盘 | 是否理解Token概念 |

**观察维度：**
- 任务完成率（是否成功）
- 迷失度（偏离最优路径的次数）
- 求助频率（主动询问"怎么办"的次数）
- 情绪曲线（任务过程中的表情/语气变化）
- 出声思维（Think Aloud的记录）

### 3.7 方法六：A/B原型测试

**目标：** 在关键设计决策点上做数据驱动的选择。

**测试场景（3组）：**

| A/B组 | 场景 | 变体A | 变体B | 评估指标 |
|:--:|------|------|------|------|
| AB1 | 对话首页布局 | 传统聊天列表+对话区 | 双栏：左侧Agent列表+右侧对话 | 首次理解时间、偏好投票 |
| AB2 | 记忆模块入口 | "记忆"独立Tab | 对话内嵌"回忆"按钮 | 发现率、使用频率 |
| AB3 | 设置引导方式 | 向导式（3步引导） | 直接进入+气泡提示 | 完成率、满意度 |

## 四、执行协议

### 4.1 时间线

```
Week 1-2 ─── 研究准备
  Day 1-3: 招募启动，筛选问卷上线
  Day 4-7: 访谈提纲定稿，原型准备
  Day 8-10: 参与者确认，排期
  Day 11-14: 预测试（2人），方法微调

Week 3-4 ─── 阶段一：探索性研究
  Day 15-22: 23场深度访谈（每天2-3场）
  Day 15-26: 10人日记研究（5天/人，滚动进行）
  Day 27-28: 阶段一数据整理，假设提取

Week 5-6 ─── 阶段二：验证性研究
  Day 29-35: 问卷投放（150-200人，3天投放+4天回收）
  Day 29-35: 竞品对比分析（同步进行）
  Day 36-42: 定量数据分析 + 假设验证

Week 7-8 ─── 阶段三：评估性研究
  Day 43-49: 10人可用性测试（每天1-2人）
  Day 43-49: A/B原型测试（与可用性测试合并执行）
  Day 50-56: 行为数据分析 + 综合洞察

Week 9-10 ── 分析与交付
  Day 57-63: 三阶段数据三角验证，撰写研究报告
  Day 64-66: 内部评审，修改
  Day 67-70: 最终交付 + 研究成果分享会
```

### 4.2 资源需求

| 资源 | 数量 | 说明 |
|------|:---:|------|
| 研究员 | 1人 | 主导全部研究设计与执行 |
| 助理/观察员 | 1人 | 可用性测试记录、访谈笔记 |
| 访谈红包 | 23×150 = 3,450元 | 深度访谈激励 |
| 日记研究红包 | 10×100 = 1,000元 | 5天日记激励 |
| 问卷抽奖 | 30×60 = 1,800元 | 15%中奖率×200人 |
| 可用性测试红包 | 10×250 = 2,500元 | 含任务完成激励 |
| **总预算** | **约8,750元** | 不含工具/平台费用 |

### 4.3 伦理与隐私

- 所有参与者在研究前签署《知情同意书》
- 录音/录屏需单独获得许可，参与者可随时要求暂停或退出
- 数据匿名化处理：报告中不使用真实姓名，使用画像编号（A1-A5等）
- 原始数据（录音/录屏/问卷）存储于本地加密环境，研究结束后90天销毁
- 不收集与研究无关的个人信息

## 五、分析框架

### 5.1 定性数据分析（访谈+日记）

**分析方法：** 主题分析法 (Thematic Analysis)

```
步骤1：转录与熟悉
  ├── 访谈录音转文字
```

### 5.2 定量数据分析（问卷）

**分析方法：**

| 分析类型 | 方法 | 工具 | 产出 |
|------|------|------|------|
| 描述性统计 | 均值/中位数/标准差/频次分布 | Python(pandas) | 各维度得分概览 |
| 群体差异 | ANOVA + 事后检验(Tukey HSD) | Python(scipy) | 5类用户差异显著性报告 |
| 需求优先级 | MaxDiff HB模型 | Python(choicemodels) | 9个Tab的无偏差优先级排序 |
| 价格敏感度 | Van Westendorp PSP分析 | Python | 最优价格区间 |
| 相关性分析 | Spearman相关系数矩阵 | Python(seaborn) | 痛点×功能期望的关联热力图 |
| NPS驱动因素 | 关键驱动因素分析(KDA) | Python | 影响NPS的关键功能Top5 |

### 5.3 可用性数据量化

**SUS量表转换：**

完成每个可用性测试任务后，收集SUS(System Usability Scale)10题量表，转换为0-100评分。

| SUS分数 | 等级 | 含义 |
|:--:|:--:|------|
| ≥80.3 | A | 优秀——不需要大幅修改 |
| 68-80.3 | B | 良好——有改进空间 |
| 68 | C | 及格线——需要针对性优化 |
| 51-68 | D | 差——需要显著重构 |
| <51 | F | 极差——需要重新设计 |

**任务完成效率公式：**

```
任务效率 = (完成率 × 满意度) / 平均完成时间

越高越优。用于对比不同任务间的体验差异。
```

### 5.4 三角验证 (Triangulation)

三种方法对同一问题的交叉验证：

```
        访谈发现          问卷数据
         "用户说"         "用户填"
              \           /
               \         /
              三角验证区
              (高可信度洞察)
               /         \
              /           \
         行为观察          竞品分析
         "用户做"         "竞品怎么做"
```

**验证规则：**
- 三角一致 → 高可信度，直接作为产品决策依据
- 两角一致 + 一角矛盾 → 中可信度，标注矛盾源，需额外验证
- 单一来源 → 低可信度，标记为"假设"，不直接驱动重大决策
- 全矛盾 → 方法论问题，回溯检查样本/问卷/任务设计
