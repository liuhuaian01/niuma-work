# 设置模块数据分析报告 — 超级牛马 v1.7

> 分析师：数析 (Metric) | 周期：2026-06-14 | 版本：v1.5 → v1.7 目标

## 方法论说明

本报告基于平台当前定性数据（太极引擎实现度 50.7%、25项功能 76% 完成率、22条后端路由、12张数据库表、12个设置面板）进行定量化推演。采用「假设驱动分析」方法：由已知结构推导指标权重，建立评分模型，输出可操作的优先级排序。所有评分均有推理链支撑。

## 一、影响矩阵分析：首次用户体验 (FTUE)

### 1.1 评估框架

定义「影响力」为该设置在用户前 5 分钟体验中的权重：
- **阻断级 (Blocking)**：不设置则核心链路不通
- **体验级 (Experience)**：影响感知质量但不阻断
- **无关级 (Neutral)**：首次使用不需要接触

### 1.2 FTUE 影响矩阵

| 设置项 | 是否阻断首次对话 | 影响用户数占比 | 对FTUE影响级别 | 推理依据 |
|--------|:---:|:---:|:---:|----------|
| **Capability Flags（能力开关）** | ✅ | 100% | 🔴 阻断 | 7种能力默认关则首条对话无法执行搜索/MCP等，默认开则Token预算可能超支；是「49:1」的核心锚点 |
| **Token Budget（令牌预算）** | ✅ | 100% | 🔴 阻断 | 默认200000，若模型上下文窗口<200K则调用失败；首条对话即触发预算分配 |
| **Model Selection（模型选择）** | ✅ | 100% | 🔴 阻断 | 无默认模型则首条消息无法路由；目前引擎实现度50.7%，Smart Allocator仅70%，回退逻辑不完整 |
| **Workspace Config（工作间配置）** | ❌ | 60% | 🟡 体验 | 首次用户无工作间概念，默认单工作间即可；但团队用户后续必用 |
| **Onboarding Flow（引导流程）** | ❌ | 100% | 🟡 体验 | 不阻断，但决定「60岁大妈能否在30秒内理解产品」 |
| **Governance Rules（治理规则）** | ❌ | 40% | 🟡 体验 | Dual-Track Governance 60%完成，可先跑默认规则 |
| **Appearance（外观偏好）** | ❌ | 30% | 🟢 无关 | 不影响核心链路，但暗色模式是开发者群体的第一印象 |
| **Notifications（通知）** | ❌ | 70% | 🟢 无关 | 首次对话无需通知，但后续体验关键 |
| **Hotkeys（快捷键）** | ❌ | 20% | 🟢 无关 | 进阶用户需求 |
| **Voice Input（语音输入）** | ❌ | 15% | 🟢 无关 | 细分场景 |
| **Security（安全环境）** | ❌ | 50% | 🟡 体验 | 本地数据不出原则要求安全设置预配置 |
| **Account Profile（个人资料）** | ❌ | 40% | 🟢 无关 | 不影响对话 |

### 1.3 核心发现

```
FTUE阻断路径 = 模型选择 → 能力开关 → Token预算
```

这三者构成「首次对话铁三角」：任一设置缺失或错误，第一条消息都无法正确执行。**这三项必须在用户打开应用前完成预配置（Auto模式），不可暴露为必须手动设置的选项。**

## 二、频率 × 影响力分析

### 2.1 双维度矩阵

以「修改频率」为 X 轴，「对产出物质量影响」为 Y 轴，划分四象限：

```
        高影响力
          │
  象限II  │  象限I
 「设置后  │  「高频+
   即忘」  │   高杠杆」
          │
──────────┼─────────── 高频修改
          │
  象限III │  象限IV
 「僵尸    │  「频繁
   设置」  │   但低影响」
          │
        低影响力
```

### 2.2 设置项落位

| 设置项 | 预估修改频率 | 影响力评分 (1-10) | 象限 | 推理 |
|--------|:---:|:---:|:---:|------|
| **Model Selection** | 低 (1-2次/生命周期) | 10 | II | 选对一次基本不变，但选错影响所有产出 |
| **Token Budget** | 低 (0-1次) | 9 | II | 默认值覆盖90%场景，进阶用户可能调高 |
| **Capability Flags** | 中 (按项目切换) | 8 | I/II边界 | 不同任务需要不同能力组合 |
| **Workspace Config** | 中 (按项目) | 7 | I | 每换一个项目可能需要调整 |
| **Governance Rules** | 低 | 6 | II | 团队管理者设置后稳定运行 |
| **Appearance** | 中 (1-2次) | 3 | IV | 个人偏好，不影响产出 |
| **Notifications** | 中 (2-3次调优) | 4 | IV | 影响体验但不影响产出质量 |
| **Hotkeys** | 低 (设置一次) | 4 | III | 不影响核心功能 |
| **Voice Input** | 低 | 3 | III | 细分功能 |
| **Security Settings** | 极低 | 8 | II | 安全是铁则第一条，但预设好不需改 |
| **Onboarding** | 极低 (仅首次) | 5 | III | 一次性，但决定留存 |
| **About/Update** | 极低 | 1 | III | 纯信息展示 |

### 2.3 关键洞察

**象限I（高频+高杠杆）是v1.7的核心设计空间**：
- Capability Flags：每次任务切换可能需要调整能力集
- Workspace Config：项目级别配置

**象限II（低频+高杠杆）需要最佳默认值**：
- Model Selection、Token Budget、Security → 这三个必须在出厂时做到「打开即用，永远不用改」

|:---:|--------|----------|:---:|------|

> 引擎实现度最低的模块（Dynamic Balancer 25%、Attention Engine 30%）正好处于依赖链的关键位置，这是v1.7最大的技术风险。

## 四、默认值决策树：49:1 天道法则

### 4.1 决策框架

```
对于每个设置项，回答三个问题：
  Q1: 用户犯错（设错值）的代价是什么？
      → 高代价 → Auto（平台守住）
      → 低代价 → 可暴露
  Q2: 用户是否有足够的上下文做出正确判断？
      → 需要专业知识 → Auto
      → 纯个人偏好 → 可暴露
  Q3: 该设置是否影响多用户/多会话的一致性？
      → 影响全局 → Auto
      → 仅影响个人 → 可暴露
```

### 4.2 49:1 分类结果

#### 平台守住 49（默认Auto/隐藏/推荐）

| 设置项 | 默认策略 | 为何是49 | 默认值 |
|--------|:---:|------|--------|
| **Model Selection** | 🔒 Auto | 用户不应关心模型选型；选错代价=整个对话质量崩塌 | 国产模型优先（低Token铁则），Smart Allocator自动选 |
| **Token Budget** | 🔒 Auto | 用户不理解Token概念（零概念铁律）；设错=对话中断 | 200000，引擎按需分配 |
| **Compression Thresholds** | 🔒 Auto | 纯技术参数，用户无判断依据 | 60/80/90/95% 阶梯式 |
| **Capability Flags：基础6种** | 🔒 Auto | 用户不应理解 fetch/search/mcp/skills/attachments/memory 的技术含义 | 全开，预算约束 |
| **Governance Rules** | 🔒 Auto | 治理规则需要系统级一致性 | 默认规则集 |
| **Dynamic Balancer参数** | 🔒 Auto | 25%完成度，仍在迭代，暴露无意义 | 引擎内部控制 |
| **Attention Engine参数** | 🔒 Auto | 30%完成度，同上 | 引擎内部控制 |
| **Self-Healing策略** | 🔒 Auto | 55%完成度，零概念铁律 | 自动恢复 |
| **Security Settings** | 🔒 Auto | 安全是不可妥协的铁则第一条 | 本地优先，加密开启 |
| **Safety Environment** | 🔒 Auto | 安全铁则 | 默认安全沙箱 |

#### 用户掌控 1（开关/提高/切换/手动选择）

| 设置项 | 暴露方式 | 为何是1 | 默认值 |
|--------|:---:|------|--------|
| **Appearance (暗/亮/跟随系统)** | 🎛️ 手动选择 | 纯个人偏好，无对错 | 跟随系统 |
| **Language** | 🎛️ 手动选择 | 个人偏好 | 系统语言 |
| **Notifications 开关** | 🔘 开关 | 个人偏好+隐私 | 关键通知开 |
| **Hotkeys 自定义** | 🎛️ 手动选择 | 效率偏好 | 预设一套 |
| **Subagents 开关** | 🔘 开关 | 这是7种能力中唯一需要用户理解的（"让AI帮你调另一个AI"），但需配Auto预算帽 | 默认关（Token消耗大户） |
| **Voice Input 开关** | 🔘 开关 | 使用场景明确 | 默认关 |
| **Account Name/Avatar** | 🎛️ 手动选择 | 身份表达 | 默认头像+随机名 |

### 4.3 统计

| 分类 | 数量 | 占比 |
|------|:---:|:---:|
| 平台守住 (49) — Auto/隐藏 | 10 | ~59% |
| 用户掌控 (1) — 暴露 | 7 | ~41% |
| **合计** | **17** | **100%** |

> **注意**：当前实际比例为 59:41，距离理念目标 49:1 (=98:2) 还有差距。但考虑到产品尚在 MVP 阶段，完全 Auto 化需要引擎实现度≥80%。当前策略是「关键路径 Auto + 偏好暴露」。

## 五、设置项定量评估框架 (FICO模型)

### 5.1 模型设计

为每个设置项建立 **FICO 评分**（0-100分），决定其设计和开发优先级：

```
FICO = w1·F + w2·I + w3·(100-C) + w4·U

其中：
  F = Frequency (使用频率, 0-100)
  I = Impact (影响力, 0-100)
  C = Complexity (实现复杂度, 0-100; 取反：越简单分越高)
  U = Urgency (紧迫性, 0-100)

权重（基于铁则优先级推导）：
  w1 = 0.15 (频率 — 高效原则)
  w2 = 0.35 (影响力 — 质量+安全原则，权重最高)
  w3 = 0.20 (逆向复杂度 — 轻量原则)
  w4 = 0.30 (紧迫性 — v1.7导向)
```

|:---:|--------|:---:|:---:|:---:|:---:|:---:|:---:|
| 1 | Model Selection (Auto) | 30 | 95 | 80 | 100 | **82.8** | P0 |

## 七、MVP 设置范围建议：v1.7 最小设置集

### 7.1 决策原则

> v1.7 只有一个目标：**让第一条真实对话链路跑通**。

## 八、总结与建议

### 8.1 三句核心结论

1. **「首次对话铁三角」(模型+Token+能力) 必须在Auto模式下零配置运行** — 这是v1.7不失败的前提。但这恰恰依赖引擎完成度最低的两个模块 (Dynamic Balancer 25%, Attention Engine 30%)。

2. **12→8→4 的收敛路径** — 当前12面板先收敛到8面板（本报告建议），v1.7 MVP 仅需4-5个面板，暴露给用户的控制不超过2个。

3. **49:1 在当前阶段是方向而非硬指标** — 引擎综合完成度50.7%意味着大量Auto能力不可靠，此时强行隐藏所有设置会因引擎故障导致产品不可用。建议v1.7采用「Auto为主 + 有限暴露」策略，随引擎成熟度提升逐步收敛。

### 8.2 风险雷达

| 风险 | 严重度 | 缓解措施 |
|------|:---:|------|
| Dynamic Balancer 仅25% — Capability Flags Auto化不可靠 | 🔴 高 | v1.7硬编码6种能力全开，不做动态平衡 |
| Attention Engine 30% — 压缩阈值可能误触发 | 🔴 高 | 提高默认阈值至95%，保守压缩 |
| Smart Allocator 70% — 模型切换可能失败 | 🟡 中 | 必选一个国产模型作为永不切换的锚点 |
| 设置存储三层架构一致性问题 | 🟡 中 | v1.7统一读写路径，加版本号 |
