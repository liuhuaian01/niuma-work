# 太极引擎吸收 Hermes 核心能力 — 功能规格书 (PRD)

> 版本: v1.0 | 作者: 析客 (需求分析师) | 日期: 2025-04-25
> 原则: "不该做的不做，该吸收的吸收" — 刘老爷

---

## 一、问题陈述

太极引擎当前通过 Hermes 管道借用 9 项核心智能能力（辅助模型路由、上下文压缩、后台审查、数据管理、记忆系统、持久记忆文件、技能创建、任务编排、多平台消息）。这种"借用"模式带来三重风险：

1. **战略风险**：Hermes 闭源或方向改变时，太极引擎的智能层会断崖式降级
2. **体验风险**：管道延迟和中间层错误放大影响终端用户体验
3. **商业风险**：无法独立销售太极引擎给不使用 Hermes 的客户

刘老爷决策：**太极引擎要凌驾于 Hermes 之上**。核心智能能力必须吸收为原生子模块，非核心能力（桌面壳、模型适配器、MCP 客户端等）继续依赖。

### PRD-001: 分流调度（Flow Router）

#### 问题陈述
Hermes 的 auxiliary_client.py（5832 行）实现了 13 种辅助任务（摘要、格式化、关键词提取、翻译等）到廉价模型的路由。太极引擎如果继续走 Hermes 管道做辅助任务路由，每次调用都增加一跳网络延迟和故障点，且路由逻辑不透明。

#### 核心能力描述
太极引擎内置的智能任务分流调度器。自动识别 13 种辅助任务类型并路由至廉价模型执行，主模型仅处理核心推理任务。内置 Provider 降级链（fallback chain），当首选廉价模型不可用时自动切换，用户无感知。目标是降低辅助任务 Token 成本 40-60%，保持可用性 ≥99.5%。

#### 吸收等级: **P0** — 立即开始自研
理由：路由决策是太极引擎的"中枢神经"，不应假手于人。自研后可直接对接太极引擎的 Provider 适配层，消除 Hermes 中间跳。

#### 用户故事
> 作为网文作者（刘老爷），我在使用太极引擎做章节大纲规划时，引擎自动把"统计本章字数""提取角色列表""格式化对话"等小任务用便宜模型处理，长篇推理任务用高端模型。月底一看账单，比上个月少了快一半——而且完全没感觉质量下降。

#### 验收标准

**AC-001-1: 辅助任务自动识别与路由**
- Given 用户发起一条包含子任务的消息（如"帮我总结这篇文章，提取关键词，并翻译成英文"）
- When 分流调度器分析消息内容
- Then 识别出"摘要""关键词提取""翻译"三个辅助子任务，各子任务被路由至对应的廉价模型执行，主任务（协调/综合判断）由主模型处理

**AC-001-2: Provider 降级链**
- Given 首选廉价模型 Provider（如 GPT-4o-mini）返回超时或 5xx 错误
- When 分流调度器在 2 秒内未收到正常响应
- Then 自动切换至次选 Provider（如 Claude Haiku），重试成功，且用户收到的最终响应不包含任何降级信息

**AC-001-3: 成本可视化**
- Given 会话结束时
- When 用户查看 Token 用量统计
- Then 显示"主模型用量 / 辅助模型用量 / 节省金额"三项指标

**AC-001-4: 路由决策可审计**
- Given 开发者查看引擎日志
- When 查询任意辅助任务的路由记录
- Then 可看到"任务类型 → 选择模型 → 实际执行时间 → 是否降级"的完整链路

#### 信息架构

```
SmartAllocator
├── FlowRouter (分流调度)          ← 本模块
│   ├── TaskClassifier             # 13种任务类型识别
│   ├── ModelSelector              # 模型选择策略
│   ├── FallbackChain              # 降级链管理
│   └── RouteAuditor               # 路由日志
├── TokenBudget (缩龙成寸)          ← PRD-002
└── CostMonitor                    # 成本监控
```

#### 与 Hermes 的关系: **完全自研**
Hermes 的 auxiliary_client.py 仅作为设计参考（路由策略、降级逻辑），不依赖 Hermes 管道。太极引擎自研分流调度器，直接对接太极引擎的 Provider 适配层。

#### Non-goal（v1 不做）
- ❌ 不支持用户自定义路由规则（v2 开放策略配置）
- ❌ 不做模型性能实时 benchmark（使用静态评分 + 定期更新）
- ❌ 不构建新的模型 Provider 适配层（继续依赖太极引擎现有适配层或 Hermes 适配器）

### PRD-002: 缩龙成寸（Context Condenser）

#### 问题陈述
长对话导致上下文窗口爆满，要么截断丢失信息，要么 Token 费用飙升。Hermes 的 context_compressor.py 实现了双层压缩（结构化摘要 + tail 保护），但压缩策略应深度绑定太极引擎的 Token Budget 体系，不应走外部管道。

#### 核心能力描述
太极引擎原生的上下文压缩器。采用双层压缩架构：第一层对历史对话生成结构化摘要（保留关键决策、任务状态、用户偏好），第二层对近期消息做选择性保留（tail 保护）。压缩后信息保留率 ≥95%，压缩比 1:5~1:10（视上下文类型），全流程嵌入太极引擎的 Token Budget 预算体系。

#### 吸收等级: **P0** — 立即开始自研
理由：上下文压缩直接影响 Token 成本和对话质量，是用户感知最强的能力之一。必须原生实现以做到零延迟、与 Token Budget 深度耦合。

#### 用户故事
> 作为全栈开发者，我和太极引擎连续讨论了 6 个小时的架构设计。对话已经滚了几万 token，但引擎依然清楚记得 3 小时前我们定下的数据库选型决策，不会突然"失忆"。而且我没被额外收超长上下文的费用。

#### 验收标准

**AC-002-1: 双层压缩触发**
- Given 当前对话上下文接近 Token Budget 设定的阈值（如 80%）
- When 引擎检测到阈值触发
- Then 自动执行第一层压缩（生成结构化摘要），摘要包含：关键决策点、进行中任务、用户显式偏好；第二层保留最近 N 轮完整对话（tail 保护），N 可配置

**AC-002-2: 信息保留率**
- Given 一段包含 10 个关键信息点（决策、数据、偏好、任务状态）的 20 轮对话
- When 缩龙成寸执行压缩至原始长度的 20%
- Then 压缩后的上下文中至少能还原 9/10 个关键信息点（信息保留率 ≥90%）

**AC-002-3: 用户偏好保护**
- Given 用户在对话中明确表达"记住：我不喜欢 JSON 格式，用 YAML"
- When 压缩发生时
- Then 该偏好被标记为"永久保留"（persistent marker），即使后续对话很长也不会被压缩丢弃

**AC-002-4: 与 Token Budget 联动**
- Given Token Budget 设定的会话上限为 100K tokens
- When 当前上下文达到 80K tokens
- Then 缩龙成寸自动触发压缩，目标压缩至 40K tokens；若压缩后仍接近上限，再次触发

#### 信息架构

```
TokenBudget
├── ContextCondenser (缩龙成寸)    ← 本模块
│   ├── LayerOneSummarizer         # 结构化摘要
│   ├── LayerTwoTailKeeper         # Tail 保护
│   ├── PersistenceMarker          # 用户偏好永久标记
│   └── CompressionEvaluator       # 信息保留率评估
├── FlowRouter (分流调度)          ← PRD-001
└── BudgetMonitor
```

#### 与 Hermes 的关系: **完全自研**
参考 Hermes context_compressor.py 的双层架构思路，但算法实现、压缩策略、Token Budget 集成完全自研。不与 Hermes 管道交互。

#### Non-goal（v1 不做）
- ❌ 不做语义向量压缩（v2 考虑 embedding-based 压缩）
- ❌ 不支持用户手动触发压缩（v1 全自动）
- ❌ 不跨会话压缩（仅单会话内）

### PRD-003: 夜巡（Night Watch）

#### 问题陈述
Agent 在执行复杂任务链时可能出现逻辑错误、遗漏步骤、偏离目标——这些错误如果在任务结束后才发现，代价巨大。Hermes 的 background_review.py 实现了异步审查 Agent 行为，但审查标准应基于太极引擎自身的行为准则。

#### 核心能力描述
太极引擎的后台异步审查守护进程。在 Agent 执行任务时，夜巡以低优先级后台进程持续审查：Agent 是否偏离目标、是否遗漏步骤、输出质量是否达标、是否违反安全约束。发现问题时分级响应——轻微问题记录日志，严重问题主动中断并请求用户裁决。审查标准与太极引擎的"无为而治"治则对齐。

#### 吸收等级: **P0** — 立即开始自研
理由：行为审查是 Agent 可靠性的最后防线。必须原生实现，审查标准需与太极引擎的治则深度绑定，不能依赖外部标准。

#### 用户故事
> 作为网文作者，我让太极引擎帮我规划 20 章的大纲。夜巡在后台发现第 15 章的角色设定和第 3 章矛盾了，主动提醒我："刘老爷，第 15 章的角色年龄和前面不一致，要不要修正？" ——我改完继续写，避免了后续大改的灾难。

#### 验收标准

**AC-003-1: 异步审查触发**
- Given Agent 完成一个子任务或达到审查检查点（每 N 个步骤，N 可配置）
- When 夜巡检测到检查点
- Then 异步启动审查流程，不阻塞 Agent 主流程继续执行后续步骤

**AC-003-2: 目标偏离检测**
- Given 用户任务目标是"分析《斗破苍穹》前 10 章的人物关系"
- When Agent 在第 8 步时开始分析"网文市场趋势"
- Then 夜巡标记为"目标偏离-轻微"，记录日志并继续观察；若后续 2 步仍未回归主目标，升级为"目标偏离-严重"，暂停 Agent 并通知用户

**AC-003-3: 分级响应**
- Given 夜巡检测到一个问题
- When 评估问题严重程度
- Then 轻微问题（如格式不规范）→ 静默记录日志；中等问题（如信息遗漏）→ 追加提示给 Agent；严重问题（如逻辑矛盾、安全违规）→ 暂停执行，通知用户裁决

**AC-003-4: 审查日志可追溯**
- Given 任何被夜巡标记的事件
- When 用户或开发者查询
- Then 可看到完整审查链路：检查点时间 → 审查类型 → 发现 → 严重度 → 响应动作 → 结果

#### 与 Hermes 的关系: **完全自研**
审查逻辑自研，审查标准基于太极引擎治则。Hermes background_review.py 仅作流程设计参考。

#### Non-goal（v1 不做）
- ❌ 不做实时审查（v1 仅在检查点审查，v2 考虑流式审查）
- ❌ 不做跨 Agent 行为对比分析
- ❌ 不自动修正 Agent 行为（仅标记和通知，修正由用户或 Agent 自身执行）

### PRD-004: 清风（Clean Breeze）

#### 问题陈述
Agent 长期运行后，工作空间、临时文件、中间数据、过期缓存会不断堆积，影响性能且消耗存储。Hermes 的 curator.py 实现了自动数据生命周期管理，太极引擎需要自己的数据管理而非依赖外部清理。

#### 核心能力描述
太极引擎的自动数据生命周期管理器。自动识别过期/冗余/临时数据，按策略清理：工作间临时文件（会话结束后 N 小时清理）、中间推理缓存（LRU 淘汰）、日志归档（超期压缩）、记忆碎片（去重合并）。清理过程透明可审计，误删可恢复。以"无为"的方式保持引擎"干净"。

#### 吸收等级: **P1** — 适配器过渡 → 自研
理由：重要但非紧急。v1 可实现基于简单规则的清理（TLL + 容量阈值），v2 升级为智能清理策略。当前可先用适配器对接 Hermes curator，同时启动自研。

#### 验收标准

**AC-004-1: 会话结束自动清理**
- Given 一个用户会话结束
- When 会话结束事件触发（用户主动结束或超时）
- Then 清风在延迟 N 小时（默认 24h，可配置）后自动清理该会话的工作间临时文件，清理前向用户发送摘要通知

**AC-004-2: 磁盘容量阈值触发**
- Given 太极引擎工作目录占用超过预设阈值（如 80% 配额）
- When 清风定期巡检检测到阈值
- Then 自动触发清理：优先清理过期缓存 → 压缩旧日志 → 去重记忆碎片。清理后磁盘占用降至 60% 以下

**AC-004-3: 误删恢复**
- Given 清风清理了一批临时文件
- When 用户在清理后 7 天内发现需要恢复某文件
- Then 可从"回收站"（软删除区）恢复，7 天后物理删除

**AC-004-4: 清理审计日志**
- Given 清风执行任何清理操作
- When 清理完成
- Then 生成审计记录：清理时间、清理类型、涉及文件数、释放空间、是否触发阈值

#### 与 Hermes 的关系: **先适配再替代**
v1 阶段（适配器）：太极引擎通过适配器调用 Hermes curator 的基础清理能力。v2 阶段（自研）：太极引擎内置清风模块，完全替代。

#### Non-goal（v1 不做）
- ❌ 不做用户内容的智能分类/标签
- ❌ 不做跨设备数据同步清理
- ❌ 不做用户行为分析驱动的预测性清理

### PRD-005: 太虚境（Memory Realm）

#### 问题陈述
用户最痛的痛点：跨会话记忆丢失。Hermes 有 8 种记忆后端（holographic, hindsight, honcho, mem0 等），但太极引擎需要自己的记忆体系——不是为了多后端而多后端，而是选择最合适的架构并原生实现。

#### 核心能力描述
太极引擎的多层记忆架构——"太虚境"。采用三层记忆模型：工作记忆（当前会话，全量上下文）、情景记忆（近期会话摘要，默认 30 天）、语义记忆（长期知识和用户偏好，永久）。后端选择：v1 用 SQLite + 向量扩展（单一后端，简单可靠），v2 可选接入外部向量数据库。记忆检索采用混合策略（关键词 + 语义相似度 + 时间衰减）。

#### 吸收等级: **P1** — 先自研核心，再扩展后端
理由：记忆能力是太极引擎的核心差异化能力。但不需要 8 种后端——v1 聚焦单一高性能后端（SQLite + vec），验证架构后再按需扩展。不与 Hermes memory plugin 生态绑定。

#### 用户故事
> 作为网文作者，我上周和太极引擎讨论过"主角修炼体系应该从斗气改成功法流"。这周我开新对话继续写大纲，太极引擎开口就说："根据上次讨论，主角的功法流修炼体系已经定了，我们就从这个方向展开。"——它没忘。这是以前用别的 AI 从没有过的体验。

#### 验收标准

**AC-005-1: 三层记忆模型**
- Given 用户完成一次会话
- When 会话结束
- Then 工作记忆中的关键信息被提取为情景记忆（摘要形式），用户显式偏好被提取为语义记忆（长期存储）

**AC-005-2: 跨会话记忆召回**
- Given 用户开始新会话，提及与 3 天前会话相关的话题
- When 太虚境检索相关记忆
- Then 返回关联的情景记忆和语义记忆，注入当前会话上下文，且召回率 ≥90%

**AC-005-3: 记忆衰减与遗忘**
- Given 一段情景记忆超过 30 天未被访问
- When 太虚境执行记忆维护
- Then 该记忆被标记为"衰减"状态，检索权重降低；超过 90 天的非关键记忆自动归档（可恢复但不主动召回）

**AC-005-4: 用户可控记忆**
- Given 用户想删除或修改某段记忆
- When 用户执行"忘记 XX"或"修正记忆 XX"
- Then 太虚境立即执行，且后续会话不再召回被删除/修正的记忆

#### 与 Hermes 的关系: **完全自研**
不依赖 Hermes 的 8 种 memory 后端。自研三层记忆架构，SQLite + vec 为 v1 唯一后端。Hermes 的多后端设计仅作为"为什么我们不需要 8 种后端"的决策参考。

#### Non-goal（v1 不做）
- ❌ 不做 8 种后端支持（v1 仅 SQLite + vec，v2 按需扩展）
- ❌ 不做图记忆/关系推理（holographic memory 的高级功能）
- ❌ 不做记忆共享/团队记忆（跨用户）
- ❌ 不做记忆加密（v1 明文存储，v2 可选加密）

### PRD-006: 铭心（Heart Inscriber）

#### 问题陈述
每次新会话，AI 都是"一张白纸"——用户需要重复介绍自己的偏好、习惯、约束。Hermes 的 MEMORY.md + USER.md 双文件机制解决了这个问题，但太极引擎需要自己的持久记忆注入机制。

#### 核心能力描述
太极引擎的持久化用户画像系统。两个核心文件：USER.md（用户个人偏好、习惯、约束，~500 tokens）和 MEMORY.md（引擎对用户的认知积累、关键决策记录，~800 tokens）。每次新会话启动时自动注入上下文，会话过程中自动更新增量变更，会话结束后合并沉淀。核心原则：小（总量 ≤1500 tokens）、准（不冗余）、活（自动更新）。

#### 吸收等级: **P0** — 立即开始自研
理由：实现简单（两个 Markdown 文件 + 自动合并逻辑），用户感知极强。是投入产出比最高的吸收项。5-8 人天即可完成。

#### 验收标准

**AC-006-1: 会话启动自动注入**
- Given 用户（已存在 USER.md 和 MEMORY.md）启动新会话
- When 会话初始化
- Then 两个文件的内容被自动注入到系统提示词中，用户无需手动说明偏好

**AC-006-2: 会话中增量更新**
- Given 用户在对话中说"以后所有代码都用 TypeScript，不要 JavaScript"
- When 铭心检测到用户偏好声明
- Then 自动更新 USER.md，追加"编程语言偏好: TypeScript（不使用 JavaScript）"，并标记更新时间

**AC-006-3: 会话结束合并沉淀**
- Given 会话中累积了多处增量变更
- When 会话结束
- Then 铭心执行合并：去重、冲突解决（新覆盖旧）、Token 总量检查（超过 1500 tokens 时触发压缩）

**AC-006-4: 用户可编辑**
- Given 用户想查看和编辑自己的记忆文件
- When 用户执行"查看我的记忆"或"编辑记忆"
- Then 返回 USER.md 和 MEMORY.md 的当前内容，用户可直接修改

#### 与 Hermes 的关系: **完全自研**
双文件机制理念来自 Hermes，但实现完全自研。不与 Hermes 共享文件格式。

#### Non-goal（v1 不做）
- ❌ 不做多用户画像（v1 单用户）
- ❌ 不做画像版本管理/回滚
- ❌ 不做画像导出/导入（v1 文件即数据，可手动复制）

### PRD-007: 自化（Self-Transform）

#### 问题陈述
用户教 AI 的技巧、偏好流程、特定工作方法——换一个会话就"忘光了"，需要重新教。Hermes 的 skill_bundles.py 实现了"技能自动创建，永不忘记"的学习闭环，这是太极引擎"越用越强"的核心。

#### 核心能力描述
太极引擎的技能自进化系统。当用户在对话中反复纠正或教授同一类操作时，自化自动将该操作提炼为可复用的技能（SKILL.md），存入技能库。下次遇到同类场景自动激活。三层触发机制：用户显式要求"记住这个做法"、系统检测到 ≥3 次同类纠正、用户高度评价某次操作后主动询问是否保存为技能。

#### 吸收等级: **P1** — 适配器过渡 → 自研
理由：技能自动创建涉及复杂的模板生成、质量验证、冲突检测。v1 可先对接 Hermes skill_bundles 的创建能力，同时自研太极引擎原生的技能模板系统和验证流程。

#### 验收标准

**AC-007-1: 显式触发创建**
- Given 用户说"记住这个做法：每次生成大纲前先检查角色一致性"
- When 自化接收到创建指令
- Then 生成 SKILL.md 文件，包含：技能名称、触发条件、执行步骤、输入输出格式。存入技能库并返回确认。

**AC-007-2: 隐式触发检测**
- Given 用户在第 3 次纠正 Agent 同一类操作（如"代码注释要用中文"）
- When 自化检测到 ≥3 次同类纠正
- Then 主动询问用户："我注意到你经常纠正代码注释的语言偏好，要不要我记住这个习惯？"——用户确认后自动创建技能

**AC-007-3: 技能自动激活**
- Given 技能库中存在"大纲生成前检查角色一致性"技能
- When 用户请求生成大纲
- Then 该技能自动激活，在大纲生成前执行角色一致性检查

**AC-007-4: 技能冲突检测**
- Given 新创建技能与已有技能存在功能重叠或冲突
- When 自化检测到冲突
- Then 通知用户冲突情况，由用户决定合并、替换或保留两者

#### 与 Hermes 的关系: **先适配再替代**
v1：适配器调用 Hermes skill_bundles 的创建/激活能力。v2：太极引擎原生自化系统完全替代，技能格式与太极引擎深度整合。

#### Non-goal（v1 不做）
- ❌ 不做技能市场/技能分享（v3）
- ❌ 不做自动技能优化/迭代（仅创建和激活）
- ❌ 不做跨用户技能迁移

### PRD-008: 经纬（Warp-Weft）

#### 问题陈述
当太极引擎同时协调多个 Agent 时，用户无法看到团队工作进度、任务状态、瓶颈。Hermes 的 Kanban 系统（SQLite 状态机 + 可视化）解决了这个问题，但任务编排属于太极引擎的调度层，长期来看应原生实现。

#### 核心能力描述
太极引擎的多 Agent 任务编排与可视化系统。"经纬"——经为任务线（纵向，任务生命周期），纬为 Agent 线（横向，Agent 负载分布）。SQLite 状态机管理任务状态流转（待分配→进行中→审查中→已完成/阻塞），前端看板可视化。核心价值：用户一眼看到"AI 团队在干什么"。

#### 吸收等级: **P2** — 继续依赖 Hermes 管道，远期自研
理由：任务编排是成熟能力，Hermes Kanban 当前足够好用。吸收的价值在于深度整合太极引擎的调度策略，但紧急度低。v1 继续依赖 Hermes Kanban，v3 自研经纬。

#### 验收标准

**AC-008-1: 任务状态流转**
- Given 用户发起一个需要 3 个 Agent 协作的复杂任务
- When 经纬接收任务拆分结果
- Then 创建 3 个子任务卡片，初始状态"待分配"，分配后流转至"进行中"，完成后流转至"审查中"

**AC-008-2: 看板可视化**
- Given 当前有 5 个任务在不同状态
- When 用户查看看板
- Then 按列显示各状态的任务卡片，每张卡片显示：Agent 名称、任务摘要、已用时间、阻塞标记

**AC-008-3: 阻塞检测与通知**
- Given 某任务因等待上游结果而卡住超过 5 分钟
- When 经纬检测到阻塞
- Then 卡片标记为"阻塞"状态，向用户发送通知"任务 X 卡住了，需要你的决策"

#### 与 Hermes 的关系: **永久依赖 Hermes 管道（v1-v2），v3 自研**
当前 Hermes Kanban 成熟可用。太极引擎自研经纬的时机应在核心智能能力（P0/P1）全部吸收完成后。

#### Non-goal（v1 不做）
- ❌ 不做太极引擎原生的看板 UI（v1-v2 依赖 Hermes Kanban UI）
- ❌ 不做复杂依赖图（DAG）
- ❌ 不做工时统计/绩效分析

### PRD-009: 万川（Ten Thousand Streams）

#### 问题陈述
用户只在 IDE/桌面端能接触到太极引擎，但在微信、飞书、Discord 等日常平台上无法触达。Hermes Gateway 已对接 21 个平台，太极引擎不应重复造轮子——但在关键平台（微信、飞书）上应有原生接入能力。

#### 核心能力描述
太极引擎的多平台消息接入层。"万川归海"——所有平台的消息流入太极引擎统一处理，响应流回各平台。v1 聚焦两个最关键平台：微信（企业微信+个人微信）和飞书。采用适配器模式：优先用 Hermes Gateway 管道覆盖 21 个平台的长尾需求，太极引擎自身维护 2-3 个战略平台的直接接入。

#### 吸收等级: **P2** — 继续依赖 Hermes 管道 + 自研关键平台适配器
理由：21 个平台全覆盖工作量巨大且大部分不紧急。策略：自研微信+飞书两个关键平台的直接适配器（确保核心用户触达），其余 19 个平台继续走 Hermes Gateway。

#### 验收标准

**AC-009-1: 微信消息接入**
- Given 用户在微信群 @太极引擎 发送消息
- When 万川通过企业微信适配器接收消息
- Then 消息被路由至太极引擎处理，响应在 5 秒内返回微信群

**AC-009-2: 飞书消息接入**
- Given 用户在飞书与太极引擎机器人对话
- When 万川通过飞书开放平台接收消息
- Then 支持多轮对话、文件接收、卡片消息响应

**AC-009-3: 平台降级**
- Given 微信直接适配器不可用（如接口限流）
- When 万川检测到故障
- Then 自动 fallback 至 Hermes Gateway 的微信管道，用户无感知

**AC-009-4: 统一消息格式**
- Given 从不同平台（微信/飞书/Discord）发来的消息
- When 万川接收消息
- Then 统一转换为太极引擎内部消息格式，引擎对平台差异无感知

#### 与 Hermes 的关系: **混合模式**
- 微信、飞书：太极引擎自研直接适配器（战略平台）
- 其余 19 个平台：继续依赖 Hermes Gateway 管道
- 长期（v3）：按需逐步增加自研平台适配器

#### Non-goal（v1 不做）
- ❌ 不做 21 个平台全覆盖
- ❌ 不做消息加密/端到端加密
- ❌ 不做平台特有的富媒体渲染（v1 仅文本+基础卡片）
- ❌ 不做消息广播/群发

## 九、开放问题

1. **铭心的 Token 上限**：USER.md + MEMORY.md 合计 ≤1500 tokens 是否合理？是否需要根据用户等级（L1/L2/L3）动态调整？
2. **太虚境的后端选择**：v1 用 SQLite + vec 扩展（如 sqlite-vec），是否有更好的轻量向量数据库选择？（LanceDB? DuckDB?）
3. **分流调度的 13 种任务类型**：Hermes 定义的 13 种辅助任务是否完全适用于太极引擎？是否需要增删？
4. **万川的微信接入方案**：企业微信 vs 个人微信（需 WeChaty 等非官方方案）？合规风险如何评估？
5. **自化的质量控制**：自动生成的 SKILL.md 如何保证质量？是否需要人工审核环节？
6. **刚柔并济的 Hermes 依赖**：确认 Hermes 确实没有对应的"双轨治理"能力？还是其 background_review + skill_bundles 的组合已部分实现？
7. **Phase 1 并行度**：4 个 P0 项是否可以并行开发？团队是否有足够人力同时推进？
