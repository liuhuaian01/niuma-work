## 📌 TL;DR（执行摘要）

- **四层稳、两层危**：FastAPI + SQLite + Svelte 5 + 模型生态（GGUF/Ollama）2年内基本不动。PyWebView 和 Agent MCP 协议适配是必须在12个月内解决的两颗定时炸弹。
- **最大的隐性风险不是技术，是单人维护**：代码量从 6,400→35,000 行的轨迹意味着12个月后进入不可持续区。太极引擎 29 模块全部实现后，改一个核心功能将跨越 15-22 个模块——单人认知负载超上限。

- **下一步**：立即启动 PyTauri 可行性评估 + MCP 协议适配层设计。安全从 D 级修到 C 级优先于引擎新模块开发。

## 2. 逐层详析

### 2.1 桌面壳：PyWebView → 🔴 必须换

#### 三个分析师的全票裁决

| 分析师 | 判断 | 核心论据 |
|:--|:--|------|
| 竞析 | 🔴 必须换 | bus factor = 1（单人维护，2443 commits 后停更风险）；PyTauri 官方声明"希望替代 PyWebView" |
| 瑞思 | 🔴 必须换 | Windows only 在创作者群体（macOS 50-70%）中直接丢掉一半核心用户；PyWebView 跨平台方案（WKWebView/WebKitGTK）靠不住 |
| 数析 | 🟡 12月内恶化 | WebView2 Chromium 更新每版本 +3-5% 内存；但 0-12 月内风险尚可管理 |

#### 替代方案对比

| 方案 | 包体积 | 冷启动 | 跨平台 | 迁移成本 | 2027成熟度 |
|------|:---:|:---:|:--:|:--:|:--:|
| **PyTauri** | 4-8MB | <1s | ✅ Windows/macOS/Linux | 中（Python 为主，少量 Rust 配置） | ⭐⭐⭐ |
| **Tauri 2.0 原生** | 3-5MB | <0.5s | ✅ + iOS/Android | 高（需 Rust 能力） | ⭐⭐⭐⭐⭐ |
| Electron | 120MB+ | 3-5s | ✅ | 低（Web 团队） | ⭐⭐⭐⭐⭐ |
| 浏览器先行（v1.7） | 0MB | 即时 | ✅ | 零 | 即刻可用 |

#### 推荐路径

```
v1.7：浏览器版本（localhost:18080）
  → 用户不需安装，跨平台天然覆盖
  → 验证产品核心价值（对话链路 + Agent 能力）
  
v1.8：PyTauri 桌面壳
  → 解决单维护者风险
  → 实现跨平台（Windows + macOS + Linux）
  → 包体积 <10MB，冷启动 <1s
  → 自然继承 Tauri 安全模型（沙箱、自动更新、代码签名）
```

#### 不做这个迁移的后果
- PyWebView 停更 → 整个桌面壳进入生命支持 → 被迫紧急迁移
- Windows only→ 创作者群体（50-70% Mac）直接放弃
- 包体积 >80MB + WebView2 捆绑 → 完全破坏"<50MB"承诺

### 2.2 后端框架：FastAPI → 🟢 肯定不变

#### 全票通过的理由

- **Python AI 生态不可替代**：LangChain 50k+ stars、500+ 集成。所有主流模型 SDK 优先支持 Python。Go/Rust 追不上。
- **FastAPI 是 AI Agent 后端的行业共识**：Claude Code、OpenClaw、CrewAI 全都用 FastAPI
- **Python 3.13 no-GIL 对 Agent 场景收益有限**（5-15%）：Agent 90%+ 时间等 I/O，不是 CPU bound

#### 唯一的轻量优化：ASGI Server 切 Granian

| Server | 性能 | 切换成本 | 备注 |
|--------|:--:|:--:|------|
| Uvicorn（当前） | 基线 | — | Python 实现 |
| **Granian** | **+30-50%** | **零代码改动** | Rust 实现，改启动命令即可 |

```
# 改动仅一行
uvicorn main:app → granian main:app
```

#### 2028 年需要注意的

| 信号 | 概率 | 影响 |
|------|:--:|------|
| Go 侵蚀 Agent 基础设施层（Google Genkit Go 1.0, 2026） | 中 | 高并发 streaming/MCP server 层可能引入 Go 微服务 |
| Litestar 替代 FastAPI | 低 | FastAPI 生态太深，2 年内不会被颠覆 |
| Python 3.15+ no-GIL 默认 | 高 | 对 CPU 密集路径（向量计算）有收益，但 Agent 主逻辑不受影响 |

### 2.3 前端方案：纯 HTML → 迁移 Svelte 5

#### 三个分析师的全票裁决：Svelte 5

| 分析师 | 判断 | 核心理由 |
|:--|:--|------|
| 竞析 | Svelte 5 首选 | Bundle size 对 React 保持 14x 优势。Runes 响应式系统天然适合实时状态 |
| 瑞思 | Svelte 5 主交互 + HTMX 辅助 | 流式对话的复杂状态需要响应式框架，HTMX 不够。设置页/列表页 HTMX 足够 |
| 数析 | Svelte 5 前端 <150KB gzipped | 远低于 1MB 目标，对包体积和加载速度零压力 |

#### HTMX 的边界（不是否定它）

HTMX 适合"文档型"应用（设置页、项目列表、文档浏览）。**不适合**流式对话的实时交互：
- 打字机效果（逐字渲染）
- 代码高亮的实时更新
- 多轮对话的引用/跳转
- 文件拖拽上传 + 进度条
- 多窗口状态同步

#### 为什么要现在决定（不是以后）

- 前端框架选型不影响当前的 v1.7（纯 HTML 先跑通对话链路）
- 但需要在 v1.7 之前完成 Svelte 5 的选型确认——因为 v1.8 的前端重构需要 3-5 周
- 纯 HTML 单体文件已达手工维护临界点（682KB，靠 Python 脚本修改）
- **早决定 = 避免迁移成本累积**

### 2.4 Agent 框架：太极引擎 → 🔴 需要 MCP 协议适配

#### 这个判断是竞析报告中风险最高的一层

**核心事实**：
- MCP (Model Context Protocol) 2025 年底移交 Linux Foundation，已成为 **AI 工具互操作的事实标准**
- Claude、Cursor、Windsurf、Copilot 全部原生支持 MCP
- 9700 万月 SDK 下载量——被类比为"AI Agent 时代的 TCP/IP"
- A2A (Agent-to-Agent, Google 2025) 专注 Agent 间协作，与 MCP 互补

**对太极引擎的具体影响**：

| 场景 | 不做 MCP 适配的后果 | 做了 MCP 适配的收益 |
|------|------|------|
| 用户问"帮我整理飞书文档" | ❌ 无法连接飞书 | ✅ MCP server 一键接入 |
| 用户问"从 GitHub 拉 PR 总结" | ❌ 无法连接 GitHub | ✅ MCP server 一键接入 |
| 开发者想用 Ollama 之外的模型 | ❌ 只能等太极引擎手动适配 | ✅ 标准协议，任何 MCP 模型 server 即插即用 |
| 2028 年新出的 AI 工具 | ❌ 全都支持 MCP，你的不支持 = 生态孤岛 | ✅ 自动追随行业标准 |

#### 太极引擎不需要放弃自研

**关键区分**：MCP 是 Agent ↔ 工具的通信协议，不是 Agent 的"编排逻辑"。太极引擎的差异化在"编排层"（记忆闭环、Token 管控、渐进式能力开放）。MCP 适配只需要一个协议转换层。

```
用户 → 太极引擎（记忆+编排+Token管控）→ MCP Client → 外部工具
                                          → MCP Server → 内部能力暴露
```

**建议策略**：
1. **0-6 月**：实现 MCP Client 适配层（太极引擎能调用外部 MCP server）
2. **6-12 月**：评估将太极引擎内部工具暴露为 MCP Server（其他 MCP 工具能调用太极引擎的能力）
3. **长期**：如果开源 Agent 框架足够成熟，将自研引擎降级为"行业特定记忆/编排模块"而非"通用引擎"

#### 用户感知影响

| 太极引擎特性 | 用户感知度 | 2028年竞争力 |
|-------------|:--:|------|
| Token 精准管控 | 低（用户无感） | 成本优势→定价优势→间接感知 |
| 渐进式能力开放 | 中（部分感知） | 与 MCP 工具发现机制重叠 |
| 记忆闭环 | **高（核心差异化）** | 唯一真正的护城河 |
| 29 模块架构 | 低（用户无感） | 宣传材料价值 > 用户价值 |

### 2.5 数据库：SQLite + LanceDB → 🟡 可能调整

#### SQLite FTS5：🟢 肯定不变

- 全球部署量最大的嵌入式数据库
- 单用户桌面应用永远碰不到 WAL 并发瓶颈（500+ 写/秒）
- 2 年内不需要换 Postgres（除非做 100 用户实时协作）

#### LanceDB：🟡 需要关注

| 风险维度 | 量化 | 时间窗口 |
|------|------|:--:|
| 社区规模小 | LLM 框架集成少于 ChromaDB | 持续 |
| 数据膨胀 | 12 个月后 20 万+向量 → 内存 +250MB | 12-18月 |
| 项目年轻 | 对比 ChromaDB（生产级）成熟度差距 | 持续 |

**替代方案评估**：

| 方案 | 成熟度 | 迁移成本 | 何时换 |
|------|:--:|:--:|------|
| **ChromaDB** | ⭐⭐⭐⭐⭐ | 中 | LanceDB 出问题时 |
| **sqlite-vec** | ⭐⭐ | 低（无额外依赖） | 2027 成熟后优先 |
| 纯 SQLite BLOB + numpy | — | 中 | <10 万向量的简单场景 |

**建议**：LanceDB 继续用，但建立数据生命周期策略（过期对话自动清理、向量 TTL、定期 optimize）。追踪 sqlite-vec 的成熟度——如果 2027 年成熟，可消除全部向量数据库依赖。

#### 如果未来需要跨设备同步

需增加 CRDT 同步层（~1500 行代码）+ 增量同步协议（~1000 行代码）。这是最难事后添加的架构设计，建议提前预留抽象层。

### 2.6 模型层：GGUF + Ollama → 🟢 肯定不变

#### 全票通过的理由

- **GGUF 是本地模型的事实标准格式**（HuggingFace 官方支持，llama.cpp/Ollama/LM Studio/MLX 全部使用）
- **Ollama 是本地模型部署的用户入口标准**（GitHub 100k+ stars，Docker-like 体验）
- **2027 年本地模型格局**：没有单一"默认模型"——用户根据任务切换（代码→DeepSeek、中文→Qwen、通用→Llama）

#### 需要做的架构预留

- **模型适配器模式**：OllamaAdapter / vLLMAdapter / llamaCppAdapter——抽象推理后端
- **运行时模型路由**：太极引擎的 Smart Allocator 根据任务类型自动选模型
- **隐私透明仪表盘**：用户清楚"这件事用的是本地模型，那件事调了云端 API"（瑞思核心建议）

## 5. 三个关键定位债（比技术债更致命）

瑞思从用户视角挖出了三个"一开始就偏了"的问题：

### 5.1 "60岁大妈"策略需要升级

| 当前定位 | 2028年应该是什么 |
|------|------|
| 界面简单到谁都会用 | 信任简单到敢放手让AI做事 |
| 卖"不用学" | 卖"不用盯" |

81%用户担心AI数据滥用，44%担心AI擅自行动。"60岁大妈"的真正差异化不是"界面上按钮少"，是"你可以放心让我替你干活，我不会偷你的数据"。

### 5.2 "数据不出本地"需要可验证

| 当前 | 2028年需要 |
|------|------|
| 口头承诺 | 部分开源 + 透明仪表盘 |

Cisco 数据：相信"本地=更安全"的用户从 90% 降到 86%。用户关心的不是"数据在哪"，是"谁能看到、用来干什么"。太极引擎的混合架构下，用户必须能看到"这事调了本地模型，那事调了云端 API"。

### 5.3 Windows only = 自断臂膀

创作者群体 macOS 使用率 50-70%。"Windows only"在 2028 年不等于"先做 Windows 再做 Mac"，等于"丢掉一半核心用户"。
