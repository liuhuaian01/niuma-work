## 📌 TL;DR（执行摘要）

- **技术栈选型本身没有致命伤**：FastAPI + SQLite + Python AI 生态是这个场景的最优解。拖后腿的不是技术选型，是工程纪律——在对话链路未通的前提下过度设计了29个引擎模块。
- **三个分析师全票通过的三件事**：① Agent 引擎从29模块砍到5-11个核心；② 前端从682KB单体HTML迁移到现代框架；③ 修复D级安全（HMAC硬编码）。
- **唯一的分歧在桌面壳**：瑞思建议v1.7直接浏览器跑、砍掉PyWebView；竞析和数析建议保留但用Nuitka优化。推荐浏览器优先，桌面壳推迟到v1.8。
- **关键决策**：v1.7的唯一正当目标是"用户双击→看到界面→打字→收到回复"。在这个目标达成之前，太极七律、Swarm编排、Honcho建模全部是空转。
- **下一步**：立即执行P0四件套——砍引擎模块、拆分HTML、修复安全、连通对话链路。

## 2. 关键分歧 & 裁决

### 分歧1：桌面壳要不要在v1.7砍掉？

| 分析师 | 方案 | 理由 |
|:--|:--|------|
| 瑞思 | **浏览器优先**（v1.7 直接跑 localhost:18080） | 消除WebView2兼容性、PyInstaller打包体积、冷提取——集中精力让对话链路先通 |
| 竞析 | **保留 + Nuitka**（短期不换，但用Nuitka替代PyInstaller） | 桌面壳已经集成可用，切换成本高。Nuitka同时解决体积、启动、反编译三个问题 |
| 数析 | **保留 + Nuitka + splash screen**（优化保留） | 量化数据表明优化后可达标：35-42MB + 1.4-1.8s冷启动 |

**裁决：推荐浏览器优先，理由有三**：
1. v1.7 的敌人是"产品不可用"，不是"安装体验"——浏览器跑起来再说，装exe是v1.8的事
2. 浏览器跑调试极快（F12直接看network/console，不用开DevTools连WebView2）
3. 浏灦器跑通了，后续PyWebView只是套个不同的前端展示层——后端API和前端代码完全不改

**但保留竞析/数析的Nuitka方案作为v1.8的桌面壳落地路径。**

### 分歧2：前端用 Svelte 5 还是 HTMX？

| 分析师 | 方案 | 体积 | 学习成本 | 适用场景 |
|:--|:--|:--:|:--:|------|
| 竞析 | **Svelte 5** | ~2KB | 中等 | 高交互页面（对话流、实时更新） |
| 瑞思 | **HTMX + Jinja2** | ~14KB | 极低 | 低交互页面（设置、文档、列表） |

**裁决：混合策略——Svelte 5 做主交互页面，HTMX 做设置/列表等低交互页**。
理由：Agent对话流需要流式输出、实时滚动、消息状态管理——HTMX的交换模型不够。但设置页、项目列表页用HTMX完全够。两者不存在技术冲突（Svelte组件和HTMX增强的HTML可以共存）。

## 3. 逐层详析

### 3.1 桌面壳层：PyWebView + WebView2

#### 竞品基准

| 框架 | 包体积 | 冷启动 | 内存 | 跨平台 | 单人维护 |
|------|:---:|:---:|:---:|:--:|:--:|
| PyWebView + PyInstaller | 45-80MB | ~3s | ~125MB | ❌ | ✅ |
| PyWebView + Nuitka | 25-45MB | ~0.85s | ~78MB | ❌ | ⚠️ |
| **Tauri 2.0** | 3-4MB | ~0.7s | ~187MB | ✅ | ⚠️需Rust |
| Electron | ~370MB | ~0.19s | ~105MB | ✅ | ✅ |
| **PyTauri v0.8** | 4-8MB | 接近Tauri | 接近Tauri | ✅ | ✅ |

#### 结论

PyWebView能工作但不是最优。**v1.7浏览器优先，v1.8评估PyTauri或Nuitka**。没有竞品用PyWebView做商业级产品（主要用于内部工具）。

#### 行动：v1.7 不做桌面壳。用户浏览器打开 localhost:18080。

---

### 3.2 后端框架层：FastAPI + asyncio

#### 竞品基准

| 框架 | 性能(req/s) | AI生态 | 单人效率 | 包体积 |
|------|:---:|:--:|:--:|:--:|
| **FastAPI (Python)** | ~14,000 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 小 |
| Fastify (Node.js) | ~65,000 | ⭐⭐⭐ | ⭐⭐⭐⭐ | 中 |
| Gin (Go) | ~70,000 | ⭐⭐ | ⭐⭐⭐ | 极小 |

#### 结论：✅ 不换。这是整个技术栈中最正确的选择。

Python 是 LLM SDK 的第一公民。Agent 场景下 API 调用的瓶颈是模型推理时间（秒级），而非框架 QPS（毫秒级）。FastAPI 的 Pydantic + 类型注解 + 自动 OpenAPI 文档是 Agent API 调试的利器。

唯一值得关注：**Granian**（Rust ASGI server，性能比 uvicorn 高 2-3倍），作为低成本性能提升选项。

### 3.3 前端方案层：682KB单体HTML

#### 竞品基准

| 方案 | 编译后体积 | 学习曲线 | 交互复杂度 | 单人效率 |
|------|:---:|:--:|:--:|:--:|
| 纯 HTML/JS | ~0KB | 零 | ⭐⭐ | ⭐⭐⭐ |
| **Svelte 5** | ~2-5KB | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| Vue 3.6 | ~23KB | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| React 19 | ~40KB | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **HTMX** | ~10KB | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |

#### 裁决：Svelte 5 主交互 + HTMX 辅助

- Svelte 5 是唯一同时满足"最小体积 + 最低认知负担 + 最高交互能力"的方案
- HTMX 可配合设置页、项目列表等低交互场景
- 从纯HTML迁移到Svelte的学习曲线是所有框架中最平缓的（.svelte文件 = script + style + html）
- 迁移工期：1-2周（单人）

#### 行动：v1.7 拆拆单体HTML（最低成本）→ v1.8 迁移到 Svelte 5

### 3.4 Agent 框架层：自研29模块太极引擎

#### 竞品基准

| 框架 | 模块数 | 单人维护 | 本地优先 | 成熟度 |
|------|:--:|:--:|:--:|:--:|
| 自研太极引擎 | 29 | ❌ | ✅ | 50.7% |
| **Claude Code** | ~5 | ✅ | ⚠️ | Production |
| **CrewAI** | ~8 | ✅ | ⚠️ | Stable |
| Kimi Work | 未知 | — | ❌ | Production |

#### 三个分析师的全票裁决

| 分析师 | 建议保留核心模块数 | 理由 |
|:--|:--:|------|
| 竞析 | **5个** | 对话管理器 + 模型适配器 + 工具执行器 + 记忆系统 + 工作区管理 |
| 瑞思 | **11个** | 5核心 + 6个已接入主链路的（TokenBudget、ChatHooks等） |
| 数析 | **5-7个**（A类模块） | 已接入主链路的模块保留，其余懒加载 |

#### 裁决：v1.7 保留 5 核心模块

```
核心5模块（v1.7）：
├─ 1. 对话管理器 — 多轮对话、上下文窗口管理、消息路由
├─ 2. 模型适配器 — 统一接口对接 Ollama/DeepSeek/Kimi
├─ 3. 工具执行器 — 文件读写、搜索、代码执行
├─ 4. 记忆系统 — 短期对话上下文 + SQLite FTS5/LanceDB 长期记忆
└─ 5. 工作区管理 — 文件树、标签页、项目上下文
```

其余24个模块全部标记为惰性加载——代码不删，但启动时不初始化，主链路不使用。

**理由**：Claude Code 用5个核心模块支撑了Production级产品。你的29个模块50.7%实现度说明：你花了一半时间在用户感知不到的地方。

#### 行动：2-3周裁剪非核心模块，先连通"打开→对话→回复"链路

### 3.5 数据库层：SQLite FTS5 + LanceDB

#### 结论：✅ 不换。

- SQLite FTS5：无可替代，零配置零维护，行业共识（Claude Code、OpenClaw、WorkBuddy全部使用）
- LanceDB：磁盘优先的向量存储设计正确，但需备选方案（ChromaDB或纯SQLite向量）
- 两者对 <50MB 目标几乎零影响

## 7. 最终裁决：是不是最优解？

### 逐层判定

| 层 | 判定 |
|------|------|
| 桌面壳 PyWebView | ⚠️ **可接受但不是最优**——没有竞品用这个做商业产品 |
| 后端 FastAPI | ✅ **最优**——Python AI 生态碾压一切 |
| 前端纯HTML | ❌ **远非最优**——已达手工维护临界点 |
| Agent 29模块 | ❌ **是最差的过度设计**——Claude Code 只需要5个模块 |
| 数据库 SQLite+LanceDB | ✅ **最优**——行业共识 |
| 安全 D级 | ❌ **最差**——硬编码密钥在任何语言都是最低级错误 |

### 总体判定：**60分（方向正确，执行偏航）**

不是"技术栈选错了"，是"把精力花在了不对的地方"。

Claude Code 用5个核心模块就支撑了Production级产品。你在对话链路还没通的情况下写了29个模块——这不是技术栈的问题，这是产品优先级错位。

**最优解不是换技术栈，是把精力从"引擎架构"拉回"第一条对话"。**
