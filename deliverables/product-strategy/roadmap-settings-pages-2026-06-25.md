## 1. 三阶段时间线评估

### 1.1 总览判断

| **前置条件可操作性** | ✅ 无前置条件 | ⚠️ 需细化 | ✅ 条件明确 |

### 1.2 阶段间硬依赖分析

  │  · 一键三连的使用频率分布      │  · 场景模式是否被分析师采纳    │
  │                              │  · 哪些 P2 项应该提前          │

**硬依赖清单**：

| 依赖 | 被依赖方 | 依赖方 | 类型 | 风险 |
|------|---------|--------|:---:|------|
| 埋点数据（设置访问率/放弃率/搜索依赖率） | v1.7 | v1.8 | 数据依赖 | 如果 v1.7 用户量不足，数据无统计显著性 |
| ≥100 真实用户 | 市场/增长 | v1.8 | 业务依赖 | 用户增长不可控，可能无限期延迟 v1.8 |
| API Key 验证基础设施 | 后端/LLM 网关 | v1.7 | 技术依赖 | 当前未打通第一条真实对话，后端可能未就绪 |
| 账户体系 | 账户系统 | v2.0 | 技术依赖 | 跨设备同步依赖持久化账户 |
| 技能引擎运行时 | 技能系统 | v1.8 | 技术依赖 | Agent 配置功能依赖技能引擎的运行时状态接口 |

### 1.3 前置条件 "≥100 真实用户" 的深度评估

**问题诊断**：
- "真实用户"的定义模糊：注册用户？周活跃用户？从第一条真实对话算起？
- 没有时间上限：如果 6 个月才到 100 用户，v1.8 是否还要等？
- 100 这个数字的统计显著性存疑：设置页的转化率、放弃率、搜索依赖率这些指标，100 用户的样本量太小，置信区间可能很宽

**建议方案**（替代简单的人数门槛）：

| 条件 | 阈值 | 为什么 |
|------|:---:|------|
| 累计真实对话数 | ≥ 5,000 条 | 设置行为与对话频次正相关，对话数比用户数更准确反映"有人在使用" |
| v1.7 上线后时间 | ≥ 4 周 | 给用户足够时间从 Onboarding 过渡到日常使用，发现二次设置需求 |
| 设置相关埋点数据完备 | 7 项风险指标均有 ≥30 个数据点 | 每个指标都能支撑基本的趋势判断 |

**决策建议**：用 AND 逻辑组合这三条，而非单纯的 "≥100 用户"。如果 8 周后仍未达标，启动"提前评审"——判断是否应该不等数据、基于定性研究直接推进 v1.8。

### 1.4 总体节奏评估

PRD 的三阶段策略在战略层面是正确的渐进式路线。核心逻辑——先验证基础假设、再铺开完整体验、最后补齐生态——符合"本地优先的 AI 工作室"的克制定位。

**需调整的点**：
1. v1.7 和 v1.8 之间应该插入一个 **v1.7.x 快速迭代窗口**（2-4 周），用于根据真实用户反馈微调 Quick Settings 和 Onboarding
2. v1.8 的"≥100 用户"门槛应用上述三元条件替代
3. v2.0 时间线建议仅作为方向性指引，在 v1.8 交付后重新评估

## 5. 利益相关者沟通要点

### 5.1 高管版（战略对齐 + 业务影响）

**核心叙事**: "设置页不是功能列表，是增长基础设施。做好设置 = 减少流失 = 更多用户走到付费转化。"

| # | 要点 | 为什么重要 |
|---|------|---------|
| 1 | **Onboarding 是留存的第一道防线**：如果用户第一天就因"看不懂设置"而流失，后面的功能再强大都没机会展示。v1.7 的 Onboarding 完成率目标 > 90%，这直接关联到次日留存率 | 设置复杂度是 AI 工具类产品的前 3 大流失原因（行业数据）；超级牛马的四类画像中小白和写作者对设置零容忍 |
| 2 | **技能 Tab 是超级牛马的护城河**：竞品无一有"技能管理 GUI + Agent 可视化配置"。v1.8 的技能 Tab 交付后，竞品需要 6-12 个月才能追平。差异化能力在设置页被"发现"的概率是 30%（技能管理访问率目标），这是自然的产品内增长 | 差异化如果藏太深等于不存在。设置页是"价值发现入口"，不只是"配置工具" |
| 3 | **分阶段投入，每个阶段都有独立的 ROI**：v1.7（10 周）→ 验证设置需求假设；v1.8（18 周）→ 建立差异化感知；v2.0（12 周）→ 补齐生态。每个阶段的可交付物都是独立可用的，不需要等到 v2.0 才有价值 | 避免"全有或全无"的大项目陷阱 |
| 4 | **最大风险不在设置页本身，在对话管道的底层依赖**：如果第一条真实对话始终未打通，设置页的 API Key 验证功能会被阻塞。需要确认 LLM 网关的时间线与 v1.7 对齐 | 这个风险超出设置团队的控制范围，需要高管协调 |
| 5 | **前置条件 "≥100 真实用户" 建议替换为三元条件**：纯用户数门槛可能无限期延迟 v1.8。建议用"累计 5,000 对话 + 4 周上线 + 30 数据点"组合条件，加上 12 周硬性时间上限 | 保持路线图的可预测性，避免团队等待 |

### 5.2 工程版（技术细节 + 时间线 + 依赖）

**核心叙事**: "渐进式架构，每阶段独立可测。配置存储层从 v1.7 就开始版本化设计，不给后续留大坑。"

| # | 要点 | 技术决策 |
|---|------|---------|
| 1 | **配置存储层：v1.7 用 JSON + 版本化 Schema，预留 v1.8 的结构化迁移路径**。修改历史栈（Cmd+Z）和冲突检测从 v1.7 就设计数据模型，而非 v1.8 重构 | 技术债最小化：存储层接口抽象为 `ConfigStore`，v1.7 实现 `LocalJSONStore`，v1.8 升级为 `VersionedConfigStore`，v2.0 增加 `CloudSyncStore`——每次只换实现，不改接口 |
| 2 | **搜索基础设施：v1.8 Sprint 1.8.1 就建好分词索引，预留拼音匹配扩展点**。即使 v1.8 首次交付不做拼音，索引结构支持多字段（名称/描述/拼音/别名），后续补成本低 | 分词索引用前缀树（Trie）结构，每个设置项的索引条目包含 `{key, label, description, tabName, aliases, pinyin}`。搜索匹配先精确再模糊，300ms 防抖 |
| 3 | **技能 Agent GUI 是最大的技术不确定性**：Sprint 1.8.4 预留 3 周（vs 标准 2 周），且有一个明确的降级方案 | 如果复杂度超预期，v1.8 的 Agent 协作规则仅支持"串行"模式，Agent 列表用简单列表而非拖拽。降级方案不影响其他 Sprint 的交付 |
| 4 | **即时生效机制：配置变更走事件总线，各模块订阅自己关心的配置键**。避免"修改一个设置 → 全局刷新"的暴力方案 | 架构：`ConfigStore` → `EventBus('config:changed', {key, oldValue, newValue})` → 各 Feature Module 的 `onConfigChanged` handler。模型切换只通知对话管理器，不通知外观模块 |
| 5 | **API Key 验证：v1.7 需要后端提供统一的 Key 验证接口**。接口返回用户语言错误（"密钥无效"而非 "HTTP 401"）。如果后端未就绪，v1.7 的 API Key 模块降级为"预填表单 + 标记为即将推出" | 接口规格：`POST /api/keys/validate`，body `{provider, key}`，response `{valid, modelList, errorMessage（用户语言）}` |

### 5.3 设计版（UX 原则 + 交互模式 + 边界）

**核心叙事**: "克制不是少做，是在正确的地方做正确的事。场景模式的交互是此次设计的最大挑战，因为没有竞品参考。"

| # | 要点 | 设计方向 |
|---|------|---------|
| 1 | **左侧垂直 Tab 是确定的导航方案，不必再探索顶部水平 Tab**。7 个竞品全部使用左侧垂直 Tab，开发者的肌肉记忆是 `Cmd+,` → 左侧列表 → 点击或搜索。唯一需要设计探索的是 Tab 图标的最终选择（建议用线性图标 + 选中填充） | 导航交互：`Cmd+,` 打开 → 搜索框自动聚焦（开发者画像） / 上次关闭的 Tab（其他画像）。Tab 顺序不可自定义（保持一致性） |
| 2 | **场景模式的交互需要单独的设计冲刺**：这是 PRD 中最复杂的交互——用户需要理解"一个快照包含模型+风格+上下文+技能集"。建议用"配方"隐喻（像保存一个咖啡配方：豆子+研磨度+水温+时间）而非"配置文件" | 交互：新建场景 → 命名 + 从当前设置继承 → 可视化展示包含的参数 → 保存后出现在切换列表中。切换时有简短过渡提示"已切换到「报告模式」" |
| 3 | **蓝点系统（修改标记）是克制设计的标志**：这个细节从 Linear 学到。蓝点要小而清晰（6px），出现在设置项标签左侧和 Tab 名称旁。恢复默认后立即消失——这种即时反馈是开发者用户最在意的"可预测性" | 视觉规格：蓝点 `#3B82F6`，直径 6px，无动画（即时出现/消失）。Tab 角标显示修改项数量 |
| 4 | **Onboarding 的设计语气：像朋友推荐，不像系统配置**。第 3 步（API Key）的"不知道这是什么？跳过"文案需要反复推敲——既要让小白安心跳过，又不能贬低 API Key 的价值（开发者用户需要它） | 关键文案测试：用"接入你的专属模型"替代"配置 API Key"，用"超级牛马内置模型已经可以处理大多数日常任务"替代"内置模型够用" |
| 5 | **空状态和边缘状态的设计不能马虎**：无 API Key 时模型列表如何呈现（Q6）？搜索无结果时如何引导？技能列表为空时如何引导到技能市场？这些"没人用的状态"恰恰是用户对产品质量的第一印象 | 无技能状态："你还没有安装技能。技能是超级牛马的超能力模块 → [发现技能]"（Q3 决策影响此交互） |

## 6. 待确认问题（Q1-Q7）建议回答

### Q1: 功能开关的范围

> "功能开关管理"的具体范围？哪些功能可以在 Quick Settings 中开关？

**建议：选项 A — 仅 AI 行为相关功能**

Quick Settings 面板中的功能开关仅包含：

| 开关 | 所属类别 | 理由 |
|------|---------|------|
| 联网搜索 | AI 行为 | 直接影响 AI 回答内容 |
| 思考过程可见性 | AI 行为 | 直接影响 AI 回答呈现 |
| 多模态输入（如图片） | AI 行为 | 能力级开关，影响对话行为 |

**排除**：UI 相关功能（动画、消息密度、代码块主题）不放 Quick Settings。这类设置属于"设好就忘"型，不应占用 Quick Settings 的有限空间。

**补充建议**：v1.8 完整设置页中增加"添加到快捷面板"功能（P2），让高级用户自定义 Quick Settings 中显示哪些开关。这样既保持 v1.7 的克制，又给 v2.0 留扩展空间。

### Q2: Onboarding 是否强制

> Onboarding 是否需要强制完成？还是可以全部跳过？

**建议：选项 A 改良版 — 可全部跳过，但有一屏"确认推荐默认"**

具体流程：
1. 每步都有"跳过"按钮和"使用推荐"按钮
2. 如果用户连续跳过 3 步，最终显示一个"确认页"：
   ```
   看起来不错！
   
   我们将使用以下推荐设置：
   · 外观：跟随系统
   · 语言：中文（自动检测）
   · AI 模型：超级牛马内置模型
   
   [调整设置]  [开始使用]
   ```
3. "开始使用" = 一键确认所有默认值，进入主界面

**理由**：
- 强制 Onboarding 与"设置是最后手段"的哲学矛盾——不应该逼用户做选择
- 但完全跳过后用户不知道默认值是什么也不对——"确认页"起到了"知情同意"的作用
- 对任何画像都友好：开发者可以快速跳过 3 步 → 确认页 → 开始使用（~5 秒）；小白可以逐步骤看说明

### Q3: 技能市场入口形式

> 技能 Tab 中的"技能市场入口"是链接跳转还是内嵌？

**建议：选项 A — 按钮跳转独立技能市场页**

具体方案：
- 技能 Tab 内放置"发现更多技能 →"按钮（非主 CTA 样式，是次级按钮或文字链接）
- 点击后跳转到独立的技能市场页面（可能是一个新窗口/新 Tab，或应用内独立路由）
- 如果用户技能列表为空，显示引导卡片："技能是超级牛马的超能力模块" + "发现技能"主按钮

**理由**：
- 设置页的定位是"管理已安装的技能"，不是"发现新技能"
- 技能市场需要独立的搜索、分类、评分体系——这些在设置页内嵌会破坏信息架构
- 独立市场页可以在未来扩展为产品内的增长引擎（推荐/热门/分类），不受设置页布局约束

### Q4: 场景模式存储位置

> "场景模式"的存储位置：本地还是云端？

**建议：选项 A — v1.8 仅本地存储**

**理由**：
- v1.8 不包含跨设备同步，场景模式云端存储没有对应的消费场景
- 强制云端存储会引入不必要的认证依赖——用户不应该为了创建场景模式而必须登录
- 与"本地优先"的产品定位一致
- 如果担心用户换设备丢失场景：v1.8 的场景模式数据包含在数据导出（P2-6）范围内。用户可以通过导出 → 导入实现手动迁移

**v2.0 升级路径**：当跨设备同步（P2-5）上线时，场景模式自然纳入同步范围，存储从本地迁移到云端。

### Q5: 设置入口命名

> 完整的设置页入口命名？

**建议：混合方案 — 入口使用"⚙ 偏好"，页面标题使用"偏好设置"**

| 位置 | 命名 | 理由 |
|------|------|------|
| 主界面齿轮图标 Tooltip | "偏好" | 避免"设置"吓到小白 |
| 系统托盘右键菜单 | "偏好..." | 与主流 macOS 应用一致 |
| 设置页窗口标题 | "偏好设置" | 比"偏好"更完整，比"设置"更温和 |
| `Cmd+,` 菜单项 | "偏好..." | 与 macOS HIG 一致（macOS 原生应用全部用"Preferences..."） |

**理由**：
- Cursor 用 "Preferences"，VS Code 用 "Settings"，ChatGPT 用 "Settings"——行业没有统一标准
- "偏好"在中文语境下比"设置"更温和：偏好 = "我喜欢这样"，设置 = "你必须配置这个"
- 瑞思的用户研究发现小白对"设置"有心理门槛——"偏好"有效降低这个门槛
- 开发者用户对"Preferences"有肌肉记忆（`Cmd+,` = Preferences），"偏好"是"Preferences"的自然中文对应

### Q6: 无 API Key 时功能呈现

> API Key 未配置时，依赖云端模型的功能如何呈现？

**建议：选项 B 改良版 — 灰色显示 + 上下文感知提示**

具体方案分三级：

| 场景 | 呈现方式 | 示例 |
|------|---------|------|
| **用户从未表示对云端模型感兴趣**（Onboarding 第 2 步选了内置模型，从未进连接 Tab） | **隐藏**云端模型选项 | 模型下拉列表中不显示需 Key 的模型，减少噪音 |
| **用户在 Onboarding 中选了云端模型但未配 Key** | **灰色显示** + Tooltip "需要配置 API Key" | 模型旁有灰色钥匙图标，点击跳转到连接 Tab |
| **用户已有过 Key 但当前过期/失效** | **红色状态** + "API Key 已失效，点击更新" | 连接 Tab 的状态指示变红，主动提醒 |

**理由**：
- 纯隐藏：用户可能不知道有这个能力
- 纯灰色显示：对从不打算用 API 的用户是噪音
- 三级策略：基于用户行为信号判断其意图，精准呈现
- 对小白友好（不展示他们不需要的），对开发者友好（他们知道去哪里配 Key）

### Q7: 拼音模糊搜索

> 设置项搜索是否需要支持模糊拼音？

**建议：选项 A — 需要支持拼音模糊搜索**

**但分阶段交付**：
- **v1.8 首版**：精确中文搜索 + 常用别名映射（"主题"→外观、"key"→API Key、"记忆"→上下文记忆长度）
- **v1.8.1 补丁**（上线后 2-4 周）：拼音模糊搜索（"zhu'ti"→"主题"、"ji'yi"→"记忆"）

**理由（为什么需要）**：
- 中文开发者的肌肉记忆：在 VS Code 中输拼音搜设置是常见行为
- 不支持的代价：用户输入"zhuti"无结果 → 以为没有主题设置 → 放弃搜索 → 手动翻 Tab → 体验差
- 竞品对比：VS Code 的中文用户社区高度依赖拼音搜索（虽然 VS Code 本身不原生支持拼音，但中文用户普遍期望）

**技术可行性**：
- 使用 `pinyin-match` 或类似库，在搜索索引中增加拼音字段
- 性能影响：40+ 个设置项的拼音索引 < 5KB，搜索延迟可忽略
- 实现成本：约 3 个前端点（Sprint 1.8.1 的搜索基础设施如预留扩展点，成本可减半）

**如果资源紧张，v1.8 首版可暂不做拼音**，但必须在搜索的"未找到"状态中提供好的降级体验（如："没有找到「zhuti」，试试搜索「主题」"）。

**RICE 洞察**：
- 默认模型选择的 RICE 分数遥遥领先——这是正确的，它影响所有用户、效果确定、工作量小

- 设置搜索的 RICE 被低估了——因为 Reach 很高（搜索是所有进设置页用户的入口），实际优先级应该更高
