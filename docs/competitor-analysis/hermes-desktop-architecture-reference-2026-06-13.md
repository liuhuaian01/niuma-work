## 二、技术栈决策

| 层级 | Hermes 选择 | 超级牛马参考建议 |
|------|------------|:--|
| **桌面框架** | Electron | ✅ **推荐 Electron** — 生态成熟，Hermes 一周完成验证了可行性 |
| **打包工具** | electron-builder/forge（推测） | electron-builder（社区更大） |
| **自更新** | 内置 in-app self-update | electron-updater |
| **UI 层** | React/Vue（推测，Web 技术） | React（生态最大）或 原生 HTML（初期） |
| **i18n** | 类型安全 i18n 层 | i18next + TypeScript |
| **进程架构** | Main/Renderer 分离 | 标准 Electron 架构 |
| **Node 版本** | ≥ 20.19 / 22.12 | Node 22 LTS |
| **远程连接** | WebSocket + OAuth | 可参考，初期可简化 |
| **搜索** | SQL 限定 + FTS5 | SQLite FTS5 |
| **命令入口** | Cmd+K 命令面板 | Cmd+K + 右键菜单 |

## 三、关键特性实现参考

### 3.1 快速安装体验

Hermes 方案：
```
下载安装包 → 一键安装 → Nous Portal 登录 → 第一条消息
```

超级牛马借鉴：
```
下载安装包 → 一键安装 → 自动检测本地模型(Ollama) → 预装 Hermes 大管家 → 第一条消息
```
目标：符合"极致简单"铁则——打开到第一条消息 <5s。

### 3.2 远程网关连接

Hermes 支持桌面应用不运行本地后端，而是通过 WebSocket 连接远程 Gateway。

**对超级牛马的价值**：
- 初期（v1.0）：本地运行一切（Ollama + 本地推理），符合"数据不出本地"
- 中期（v1.5+）：可选远程 Gateway，降低用户硬件门槛
- 参考点：Hermes 的 Profile 系统支持同一窗口内多 Profile 并发——这对多工作间架构有直接参考价值

### 3.3 类型安全 i18n

Hermes 的 i18n 设计：
- 类型安全层：编译期检查翻译完整性
- 覆盖范围：聊天/侧边栏/设置/命令中心/cron/消息/profiles/skills/agents
- 切换方式：Appearance 设置 → display.language 持久化

**超级牛马借鉴**：
- 初期（v1.0）：中文优先，不引入 i18n 复杂度
- 中期（v1.5+）：引入 i18n，但可以简化——超级牛马不需要覆盖"Discord/Telegram 等消息平台"

### 3.4 模糊搜索模型选择器

四端统一（桌面/Web/TUI/CLI），输入"v4fl"即可定位 `deepseek-v4-flash`。

**超级牛马借鉴**：模型搜索是刚需，但超级牛马不需要 TUI/CLI 端，只需桌面+Web。

### 3.5 技能体系瘦身 + 门控

Hermes 的 `environments` frontmatter 门控机制：
- 上下文特定技能不会出现在无关用户的技能索引中
- 但用户主动请求时仍可加载

**超级牛马借鉴**：太极引擎可以借鉴"上下文门控"——根据当前工作间/任务类型，只加载相关技能，减少 Token 消耗。

### 3.6 性能优化细节

| 优化项 | Hermes 实现 | 节省 | 超级牛马应用 |
|--------|-----------|:--:|------|
| read_file 紧凑行号 | 减少行号冗余 | -14% Token | ✅ 文件面板语义分组的通用优化 |
| FTS5 段合并 | VACUUM 时合并 | 状态文件缩小 | ✅ 知识库索引维护 |
| 模型目录刷新 | 每小时（原每日） | 更及时的模型发现 | 🟡 超级牛马模型数量少，无需高频刷新 |
| /undo 命令 | 回退最近N轮 | 减少冗余对话 | ✅ 符合"极致体验"铁则 |

## 四、超级牛马桌面化技术路线建议

### Phase 1：原型桌面壳（1-2 周）

```
技术选型：Electron + 嵌入现有 HTML 原型
目标：将 niuma-ui-v4.0-soft-tech.html 嵌入 Electron 窗口
```

- 利用 Electron 的 `BrowserWindow.loadFile()` 直接加载已有原型
- 无需 React/Vue，纯 HTML/CSS/JS 即可
- 自更新和打包留到 Phase 2
- 验证：桌面壳的键盘快捷键、窗口管理、托盘图标

### Phase 2：原生桌面体验（2-3 周）

```
技术选型：Electron + React（或保持纯HTML）
目标：原生桌面交互（拖拽文件、通知、快捷键、自更新）
```

参考 Hermes 的关键路径：
- `apps/desktop/` 目录结构（Main/Renderer 分离）
- Cmd+K 命令面板
- 拖拽文件到聊天区域
- 状态栏模型选择器
- electron-updater 自更新

### Phase 3：远程后端（v1.5+）

```
技术选型：Electron + WebSocket
目标：可选远程 Gateway，降低硬件门槛
```

## 五、Hermes 架构中不适合借鉴的部分

| 不适合借鉴 | 原因 |
|------|------|
| CLI/TUI 双终端 | 超级牛马不需要终端界面，桌面+Web已够 |
| 消息平台矩阵 | Discord/Telegram/Slack 等非超级牛马 MVP 范围 |
| Docker-in-Docker 后端 | 超级牛马不需要容器化沙盒 |
| Kanban goal_mode | 超级牛马有自己的 Smart Allocator |
| 社区贡献者模型 | 超级牛马初期单人开发，不需要社区贡献流程 |
| 技能市场繁重体系 | 超级牛马的技能体系更精简，不需要 marketplace |

## 六、风险提示

1. **Electron 体积**：基础 Electron 包 ~100MB+，需通过裁剪（asar 压缩、移除 DevTools）满足"<50MB"铁则
2. **内存占用**：Electron + Chromium 基础 ~80MB，加上推理后端可能超"<200MB"铁则
3. **开发效率 vs 极致轻量**：Electron vs Tauri（Rust，更轻）——Tauri 体积 ~5MB 但 Rust 开发成本高
4. **自动更新签名**：macOS 需要 Apple Developer 账号 ($99/年)，Windows 需要代码签名证书

## 七、结论

**Hermes Desktop 的 1 周极速构建验证了 Electron 桌面化的可行性，但单兵作战需 2-3 周。**

超级牛马桌面化的最高优先级路径：
1. **Electron 壳嵌入现有 HTML 原型**（最快，1周可出 MVP）
2. **渐进式替换为原生交互**（拖拽/快捷键/通知，2-3周）
3. **评估 Tauri 作为长期方案**（体积优势，v2.0+ 考虑）

Hermes 最值得借鉴的三点：**快速安装体验 + Cmd+K 命令面板 + 技能门控机制**。
