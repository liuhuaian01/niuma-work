# Kimi Work Beta 竞品拆解报告

> 拆解日期：2026-06-13 | Beta 公测日期：2026-06-03

## 一、产品定位

| 维度 | Kimi Work Beta | 超级牛马 |
|------|:---:|:---:|
| 定位 | 通用型本地 Agent 工作台 | AI Agent 全能工作台 |
| 目标用户 | 知识工作者（非程序员也） | 开发者和重度 AI 用户 |
| 核心理念 | "Vibe Working" — 从写代码到干活 | "太极引擎" — 不烧Token、不打扰、不越界 |
| 交互界面 | 桌面 GUI（Mac/Windows） | 桌面应用（原型阶段，单文件HTML） |
| 运行环境 | 本地桌面客户端 | 本地桌面应用 |
| 内核模型 | Kimi K2.6 | DeepSeek V3.2 + 多模型混合 |

## 三、核心能力拆解

### 3.1 300 子 Agent 并行协作

| 维度 | Kimi Work | 超级牛马 |
|------|-----------|----------|
| 并行数量 | **最多 300** | Swarm 拓扑（Root→Worker→Gate Validator→Gate Synthesizer） |
| 创建方式 | **自动拆解**→动态创建 | 用户定义 + 模板创建 |
| 调度策略 | 自动（未公开细节） | 设计阶段（Gate Validator 门控） |
| 通信方式 | 未公开 | 经 Hermes 大管家中转 |

**差异点**：Kimi Work 的 300 Agent 是"自动创建+自动销毁"，超级牛马的 Swarm 是"用户定义+持久化"。前者适合一次性大规模并行任务，后者适合长期运行的团队协作。

### 3.2 13 小时长程任务

这是 Kimi Work 最突出的差异化能力。

| 维度 | 数据 |
|------|------|
| 连续运行 | 13 小时 |
| 工具调用 | 4000+ 次自主调用 |
| 底层模型 | Kimi K2.6 |
| 实现方式 | 推测：检查点+状态持久化+本地运行不中断 |

**对超级牛马的启示**：超级牛马的自动化任务（如每日研究）目前依赖 WorkBuddy 平台调度。Kimi Work 证明"桌面端长程 Agent"是可行且有价值的方向。超级牛马需要建立自己的长任务机制——太极引擎的"生生不息"闭环可以承载。

### 3.3 Skill 自动封装

| 维度 | Kimi Work | 超级牛马 |
|------|-----------|----------|
| Skill 创建 | **自动提炼经验→封装 Skill** | 人工编写 + 社区吸纳 |
| Skill 调用 | 安装后直接调用 | 三层生态（吸纳/自开发/自进化） |
| Skill 存储 | 本地 Skill 库 | 本地 + 社区（计划） |

**差异点**：Kimi Work 的 Skill 自动封装是超级牛马 Skill 三层自生态的参考方向。目前的超级牛马 Skill 体系仍偏人工，Kimi Work 证明"从执行经验自动生成 Skill"在技术上可行。

### 3.4 Kimi WebBridge — 浏览器操控

Kimi Work 的差异杀手锏：可登录用户订阅的付费网站，直接在浏览器中操作。

**超级牛马当前状态**：无浏览器操控能力。

**建议**：浏览器操控不属于超级牛马 MVP 范围，但应在 Roadmap 中预留接口位置。通过 MCP 协议接入 Playwright 等浏览器工具是低成本实现路径。

### 3.5 专业数据库

Kimi Work 内置金融、科研、法律专业数据库。

**超级牛马的影响**：超级牛马的知识库（RAG）是差异化方向。Kimi 靠内置数据，超级牛马靠用户自有数据——这在企业场景中是优势（隐私安全），但在消费场景中是劣势。

## 四、与超级牛马的核心差异矩阵

| 维度 | Kimi Work Beta | 超级牛马 | 超级牛马优势？ |
|------|------|------|:--:|
| **模型策略** | Kimi 自有模型（封闭） | 多模型混合（开放，本地优先） | ✅ 更灵活 |
| **记忆系统** | ❌ 无 | ✅ 记忆向量检索+三层记忆 | ✅ 核心优势 |
| **知识库** | 内置专业数据库 | 用户自有数据RAG | ✅ 企业场景优势 |
| **Token管控** | 模型内优化（-30%） | 太极引擎全链路管控（计划97%） | 🟡 待验证 |

## 五、Kimi Work 的弱点（超级牛马应该抓住的突破口）

### 5.1 无记忆系统
Kimi Work 描述中完全未提及"记忆"或"知识库"能力。每次会话独立，无法跨会话积累经验。

→ **超级牛马的记忆向量检索+三层记忆+Honcho用户建模 是核心护城河。**

### 5.2 模型封闭
仅使用 Kimi 自有模型，用户无法切换。

→ **超级牛马的多模型混合策略（DeepSeek+Qwen+Gemma+API可选）更灵活。**

### 5.3 无用户知识库
内置数据是通用数据，不是用户自有数据。

→ **超级牛马的 RAG 知识库（用户喂入+企业文档+项目上下文）在企业场景不可替代。**

### 5.4 产品过于"万能"
定位"什么都能做"可能导致什么场景都不精。

→ **超级牛马聚焦 Agent 开发者和重度 AI 用户，定位更精准。**

### 5.5 无 Linux 支持
仅 Mac/Windows，缺失开发者核心平台。

→ **超级牛马应优先支持 Linux+Mac+Windows 三平台。**

## 六、战略建议

### 竞争策略：错位竞争，不正面刚

```
Kimi Work 优势区      超级牛马优势区
  ┌─────────┐          ┌─────────┐
  │自动拆解   │          │记忆系统   │
  │300 Agent │          │知识库RAG  │
  │浏览器操控 │  ────▶  │多模型混合  │
  │内置数据   │  错位   │安全隐私   │
  │Skill自动  │          │太极引擎   │
  └─────────┘          └─────────┘

  共同战场（需要差异化）：
  • Token 效率 → 太极引擎全链路 vs Kimi 模型内优化
  • Agent 编排 → Hermes 门控 vs Kimi 自动调度
  • 桌面端体验 → 极简克制 vs Kimi 功能完备
```
