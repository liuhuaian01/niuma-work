## 1. 产品概述

### 1.1 一句话定义

**超级牛马工作台 = 太极引擎驱动的 AI Agent 协作平台。**

底层核心引擎：太极引擎（Taiji Engine）——七律驱动的自动进化系统。
底层框架：Hermes Agent v0.16.0（MIT 开源，170+ 贡献者，542 PR，874 提交的工业级 Agent 框架）。

用户通过自然语言与 Hermes Agent（平台大管家）对话，后者作为 Root Node 自动编排 Swarm 拓扑中的多个 Sub-Agent 完成复杂任务。本地优先，云端可选。独有 Token 管控体系——能力开关 + Agent 维度预算 + 降级链——竞品无一具备。

### 1.2 核心价值主张

| 维度 | 传统 AI 工具 | 超级牛马 |
|------|:----------:|:--------:|
| 部署 | 云端 SaaS，数据上传 | **本地优先，云端可选** |
| 交互 | 单轮对话，复制粘贴 | **Hermes 编排 Swarm，多 Agent 协作** |
| 记忆 | 无状态，每次从零开始 | **三级记忆系统，越用越懂你** |
| Token 管控 | 无控制，事后看账单 | **能力开关 + 预算 + 降级链，全链路管控** |
| 扩展 | 插件市场，手动安装 | **MCP 工具接入，Agent 自主发现** |
| 成本 | 按 Token 计费，不可控 | **国产模型优先，双层压缩 + 提示词缓存** |

### 1.3 产品边界（Pi 原则 —— 不该做的一律不做）

**前期不做**：
- 用户自定义 Agent 可视化编排
- 技能市场（社区分享）
- 云端同步/多设备协作
- 移动端 App
- FTP/云存储挂载

**前期聚焦**：
- RAG 知识库 → 记忆向量检索 → 上下文压缩 → MCP 工具接入

## 2. 核心设计哲学

### 2.0 太极产品哲学（根基）

超级牛马的产品 DNA 源于中国道家哲学——"道生一，一生二，二生三，三生万物"的生成论。

| 文档 | 内容 |
|:--|:--|
| `deliverables/product-strategy/taiji-daosheng.md` | 生成论——道→一→二→三→万物 完整映射 |
| `deliverables/product-strategy/taiji-engine-architecture.md` | 引擎层——七律→七引擎 架构设计 |
| `deliverables/product-strategy/taiji-philosophy.md` | 哲学层——七律总纲与产品映射 |

**道 = "更聪明地工作，而非更费力"** ——这是所有产品决策的本源判断标准。

| 原理 | 产品落地 |
|:--|:--|
| 阴阳平衡 | 本地+云端双模 · Hermes全平台权限+被动行使 |
| 四两拨千斤 | Token 管控体系——不是加资源，是精准用力 |
| 以静制动 | Hermes 只在 5 种情况说话 · Swarm 静默编排 |
| 无为而治 | Pi 准则克制设计 · 前期不做列表 |
| 顺势而为 | 站在 Hermes v0.16.0 + OpenClaw + 开源模型生态肩上 |
| 刚柔并济 | Capability Flags 硬开关 + 记忆闭环软积累 |
| 生生不息 | Hermes 闭环学习——越用越强 |

### 2.1 11 条产品设计开发铁则

> 违反任意一条，视为设计 Bug。Pi 准则兜底——不该做的一律不做，只做该做的。

| # | 原则 | 一句话 | MVP 度量标准 |
|:--:|:--|:--|:--|
| 1 | **极致安全** | 数据不出本地 | Pi 准则覆盖率 100%，API Key 本地加密 |
| 2 | **极致性能** | 快到你感觉不到 | 冷启动 < 1.5s，TTFB < 200ms，页面切换 < 100ms |
| 3 | **极致轻量** | 小、省、不打扰 | .exe < 50MB，内存 < 200MB，零常驻进程 |
| 4 | **极致 Token 低消耗** | 每一分钱花刀刃上 | 国产模型 100% 可用，上下文压缩率 50%+ |
| 5 | **极致高效** | 最少步骤完成任务 | 打开到发出第一条消息 < 5s，键盘可达率 100% |
| 6 | **极致简单** | 奶奶都能用 | 首次使用 -> 第一条对话 < 60s |
| 7 | **极致质量** | 交付物可靠 | 每个功能完整闭环，不交付半成品 |
| 8 | **极致审美** | 干净不脏 | 5 色克制色板，动画 60fps |
| 9 | **极致体验** | 无打断感 | 反馈 < 100ms，一切自然流动 |
| 10 | **极致交互** | 所见即所得 | 直接操作不中转，弹窗最小化 |
| 11 | **极致克制（Pi 原则）** | 不该做的坚决不做 | 克制是最高级的设计 |

### 2.2 冲突仲裁规则

当铁则冲突时，按此顺序裁决：

```
1. 安全 — 数据泄漏不可逆
2. 性能 — 慢 0.5 秒用户就走了
3. 轻量 — 单文件零依赖是硬约束
4. 低 Token — 国产模型优先，上下文自动压缩
5. 高效 — 先让用户上手，再让他们快
6. 简单 — 引导 60s 内完成
7. 交互 = 质量 — 两者同等，一个差则全差
```

### 2.3 编排师三条铁律（融入前端交互）

| 铁律 | 含义 | UI 实现 |
|------|:--|:--|
| **零概念** | 用户不需要知道 Director Agent、工作间切换等概念 | 小助手永远存在，进入工作间自动变 Director。侧栏只显示用户自定义名字 |
| **不确认** | 不弹窗问"确定要执行吗？"——默认执行，只拦截危险操作 | Pi 准则 balanced 默认，打断率 < 5% |
| **陪伴是风格** | SuperNiuma 只在 5 种情况说话（破冰/进度/预警/错误/总结），其余沉默 | 进度指示：一行居中淡灰文字。@补全：输入框内联浮现，Tab 确认，纯键盘 |

**SuperNiuma 说话的 5 种情况**：

| 情况 | 示例 | 触发条件 |
|------|------|:--|
| 破冰 | "嗨，我是 XX。今天想做什么？" | 首次使用 / 每天首次打开 |
| 进度 | "正在分析市场趋势..." | Agent 执行耗时任务时 |
| 预警 | "此任务预计消耗 ¥12，是否继续？" | Token 消耗超过阈值 |
| 错误 | "抱歉，刚才的分析结果有误，已重新计算" | 质量门禁 FAIL |
| 总结 | "已完成。要点：1... 2... 3..." | 任务完成后 |

## 4. 产品架构

### 4.1 整体架构图

完整可视化架构图：`deliverables/architecture-diagram.html`

> 深紫 = Hermes v0.16.0 基座 | 珊瑚 = Token 管控层 | 浅紫 = 平台核心 | 蓝 = 记忆/扩展 | 青绿 = 模型层 | 虚线 = v2.0 规划

**六层架构**：

| 层级 | 核心组件 | 说明 |
|------|------|------|
| **基座层** | Hermes Agent v0.16.0（Root Node） | 预装运行，闭环学习，Desktop+Web Dashboard+技能生态 |
| **管控层** | Token 管控（超级牛马独有） | 能力开关、Agent 预算、降级链、外网访问控制 |
| **编排层** | Swarm 拓扑 | Root → Worker → Gate Validator → Gate Synthesizer |
| **平台层** | 前端 UI + 后端服务 + 安全引擎 + 文件&模型 | FastAPI + SQLite + LanceDB + Pi 准则 |
| **记忆层** | 三级记忆系统 | L1 会话 + L2 短期档案(FTS5) + L3 知识库(LanceDB) |
| **模型层** | 预装 + 广场 + 云端 API | 本地部署(Ollama/llama.cpp) + 云端可选 + 降级链 |

### 4.2.1 工作间隔离机制

每个工作间是一个**独立沙盒**，物理隔离：

| 资源 | 隔离方式 | 说明 |
|------|:--:|------|
| 任务 | 工作间内独立队列 | WS-1 的任务不影响 WS-2 |
| 记忆 | 工作间作用域 | Sub-Agent 的记忆仅在工作间内有效 |
| 文件 | 工作间目录隔离 | 每个工作间有独立的文件空间 |
| Agent 通信 | 同工作间内直接通信 | 跨工作间必须经 Hermes 中转 |
| 上下文 | 独立对话上下文 | Sub-Agent 不可访问其他工作间的对话历史 |

### 4.2.2 Hermes Agent 全平台权限

Hermes Agent（平台大管家）拥有**全平台穿透权限**，但**被动行使**：

| 能力 | 说明 | 触发方式 |
|------|------|:--|
| 查看任意工作间 | 可查阅所有工作间的任务进度、对话记录 | 用户明确要求时 |
| 与任意 Sub-Agent 对话 | 可直接与任何工作间内的 Agent 通信 | 用户指令需要跨工作间协调时 |
| 查看所有文件 | 可访问平台上所有文件 | 用户要求全局搜索/汇总时 |
| 跨工作间编排 | 协调多个工作间 Agent 协作 | 任务涉及跨域知识时 |
| 记忆汇总 | 从所有工作间中提取共性问题 | Hermes 闭环学习时自动 |
| **不主动干涉** | 平时沉默，不代替工作间内 Agent 做决策 | ← 这是铁律

### 4.3 Swarm 编排流程

```
用户指令 → Hermes Agent (Root Node)
              │
              ▼
         解析意图 → 分解子任务树
              │
      ┌───────┼───────┐
      ▼       ▼       ▼
   Worker   Worker   Worker  ← 并行执行子任务
      │       │       │
      ▼       ▼       ▼
  Gate Validator ← 验证输出质量
              │
              ▼
  Gate Synthesizer ← 合成最终结果
              │
              ▼
         Hermes Agent → 交付用户
```

### 5.4 广场（模型市场）

**定位**：模型和技能的分发中心，区分预装与可选下载。

**内容分区**：

1. **预装模型区（已内置，开箱即用）**
   - **Gemma-4**（绿色，Google 最新开源模型）— 已安装
   - **DeepSeek V3.2**（紫色，671B MoE，128K 上下文）— 已安装
   - **Qwen 多模态**（蓝色，阿里通义，图文理解）— 已安装

2. **模型广场（用户自选下载安装）**
   - Llama 4 Scout / Mistral 3 Small / Yi-1.5 / GLM-4-9B / MiniCPM-V / InternLM3

### 5.5 文件面板

**定位**：Agent 的操作台，不是文件浏览器。语义驱动、搜索优先、任务导向。

**架构：5 个语义分组（不做通用文件树）**

```
文件面板
├── 模型        ← data/models/*.gguf, *.safetensors
├── 导出        ← data/exports/*.md, *.json, *.html
├── 日志        ← data/logs/*.log, *.txt
├── 知识库      ← LanceDB 索引 + 文档元数据
└── 配置        ← data/config/*.json, *.yaml, *.toml
```

**核心功能**：
- 搜索 + 排序（名称/时间/大小）
- 单击预览（MD/JSON/日志/图片/CSV）
- 右键 Agent 操作（发送给 Agent/复制路径/导出/删除）
- 磁盘使用概览

**不做的事**：
- Git 状态装饰器
- 文件夹新建/重命名
- FTP/云存储挂载

### 5.10 安全中心

**Pi 准则分级**：

| 级别 | 名称 | 行为 | 适用场景 |
|:--:|:--|:--|:--|
| Strict | 严格 | 所有操作需确认 | 高危环境 |
| Balanced | 均衡（默认） | 自动执行常规操作，拦截危险操作 | 日常使用 |
| Permissive | 宽松 | 最小干预，仅拦截明显危险 | 可信环境 |

**安全指标**：
- 数据不出本地
- API Key 本地加密存储
- 审计日志（操作记录、异常检测）
- Promptware 防御（三关拦截 Brainworm 攻击）

go(page) {

## 9. 质量门禁与测试策略

### 9.1 质量门禁

每步 Agent 产出必经校验，三级决策：

```
Agent 产出 → [质量门禁] → PASS → 流式输出给用户
                         → FAIL → 记录原因，通知用户，停止
                         → REWORK → 自动重试（最多 3 次）
```

**REWORK 的 UI 表现**：一行淡灰居中文字"正在优化..."，不打断对话流。3 次失败后降级为 PASS_WITH_WARNING 并标记。

### 9.2 测试策略

| 测试类型 | 覆盖范围 | 目标 |
|:--|:--|:--|
| 单元测试 | 后端 API、模型适配层、安全引擎 | 覆盖率 80%+ |
| 集成测试 | 前后端通信、Agent 编排流程 | 核心路径全覆盖 |
| E2E 测试 | 首次引导、对话流程、页面切换 | 关键用户旅程 |
| 性能测试 | 冷启动、TTFB、内存占用 | 达标 11 条铁则 |
| 安全测试 | Pi 准则、API Key 加密、审计日志 | 零漏洞 |
| 无障碍测试 | 键盘导航、屏幕阅读器、对比度 | WCAG AA |

### 9.3 Bug 密度目标

- 核心路径：0 阻塞级 Bug
- 一般功能：Bug 密度 < 0.5/千行
- 视觉回归：每次迭代前必做
