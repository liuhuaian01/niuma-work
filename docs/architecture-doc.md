## 0. 太极引擎（设计哲学层）

> 太极引擎是超级牛马工作台的底层运转机制。
> 哲学层：`taiji-daosheng.md`（道→一→二→三→万物 生成论）
> 引擎层：`taiji-engine-architecture.md`（七律→七引擎 架构设计）

### 0.1 生成论总纲：道→一→二→三→万物

| 生成阶段 | 道家概念 | 架构映射 | 说明 |
|:--:|:--|:--|:--|
| 0 | **道** | "更聪明地工作，而非更费力" | 所有决策的本源判断标准 |
| 1 | **一（太极）** | 太极引擎 | 自平衡·自优化·自进化的统一体 |
| 2 | **二（阴阳）** | 部署阴阳/行动阴阳/治理阴阳/空间阴阳 | 四象动态均衡，不是 if-else 选边 |
| 3 | **三（三元）** | 平台三元/记忆三元/管控三元/编排三元 | 从二元互动中自然涌现的稳定结构 |
| ∞ | **万物** | 所有具体功能（工作间/群聊/Skills/MCP/...） | 从三元中生出，不是设计出来 |

### 0.2 系统四象阴阳

| 阴阳对 | 阴 | 阳 | 均衡机制 |
|:--|:--|:--|:--|
| 部署 | 本地 Ollama | 云端 API | Dynamic Balancer——任务度×隐私度×预算 |
| 行动 | Hermes 沉默观察 | Hermes 主动介入 | Attention Engine——7种情况阈值触发 |
| 治理 | 刚：硬开关/预算/隔离 | 柔：记忆/偏好/降级链 | Dual-Track——刚柔互转 |
| 空间 | 工作间沙盒隔离 | Hermes 全平台穿透 | 必要性评分≥阈值才穿透 |

### 0.3 四大三元稳定结构

| 三元 | 一 | 二（阴） | 二（阳） | 三（涌现） |
|:--|:--|:--|:--|:--|
| 平台 | Hermes | Swarm 编排 | Work间沙盒 | 群聊（跨工作间协作） |
| 记忆 | L1 会话 | L2 短期档案 | L3 知识库 | Honcho 用户建模 |
| 管控 | 能力开关 | Token 预算 | 降级链 | 执行过剩为零 |
| 编排 | Root Node | Worker | Gate Validator | Gate Synthesizer |

### 0.4 七律→七引擎（表层映射）

| 太极原理 | 引擎组件 | 核心机制 | 落地阶段 |
|:--|:--|:--|:--:|
| 阴阳平衡 | Dynamic Balancer | 本地←→云端 · 沉默←→介入 · 隔离←→穿透 动态均衡 | 📋 v2.0 |
| 四两拨千斤 | Smart Allocator | 力点探测→精准分配→执行监控→降级/提升 | 🔴 v1.6 |
| 以静制动 | Attention Engine | 全局感知→7种情况主动说话 | ✅ v1.5 / 📋 v2.0 |
| 无为而治 | Self-Healing Loop | 自愈→学习→预防 | 📋 v2.0 |
| 顺势而为 | Leverage Architecture | 借力 Hermes+OpenClaw+MCP+开源模型 | ✅ 持续 |
| 刚柔并济 | Dual-Track Governance | 硬轨道(Capability/Token/Pi) ↔ 软轨道(记忆/偏好) | ✅ v1.5 / 📋 v2.0 |
| 生生不息 | Evolution Loop | 执行→沉淀→反思→传导→进化→再执行 | 📋 v2.0 |

### 0.5 太极引擎与 Hermes Agent 的关系演化

| 阶段 | 关系 | 架构形态 |
|:--|:--|:--|
| 当前 | **寄生** — 太极引擎 ⊂ Hermes Agent | 太极引擎住在 Hermes 的壳里 |
| v2.0 | **共生** — 太极引擎 ↔ Hermes Agent | 太极引擎独立运转，Hermes 作为默认运行时 |
| v2.5+ | **容纳** — 太极引擎 ⊃ Hermes Agent | 太极引擎是壳，Hermes 变成可选运行时之一 |

**不取代 Hermes——反转嵌套关系。** 当 Hermes 的架构假设开始限制太极引擎进化时，自研 Taiji Native Runtime。

### 0.6 三层持续进化机制

| 层次 | 名称 | 粒度 | 触发 | 状态 |
|:--|:--|:--|:--|:--:|
| 1 | 自愈进化 | 秒-分钟 | Gate FAIL / Token预警 / Pi拦截 / 工具去重 | 📋 v1.6 |
| 2 | 内部进化 | 分钟-小时 | 每次任务完成 → 五维进化（记忆/技能/编排/模型/安全） | 📋 v2.0 |
| 3 | 外部进化 | 天-周 | 每日研究/竞品分析/架构审查 → 自动评估 → 生成变更建议 | ✅ 已启动 |

**闭合链路**：研究发现 → Smart Allocator 评估 → 生成架构变更建议 → 审批 → 自动更新文档/配置/代码。

### 0.7 本地学习 vs 社区吸纳

```
阴（本地）：元数据驱动，不看内容。Agent 效果数据留在本机，太极引擎本地学习。
阳（社区）：Skills、开源工具、最佳实践积极吸纳。进入后做力点探测——不好用的自动降权。
```

### 0.8 天道法则与用户权限（大道五十·天衍四十九·人遁其一）

| 维度 | 平台守护（四十九） | 用户掌控（一） |
|:--|:--|:--|
| 外网访问 | 默认关 · 自动检测本地答案 · 预估消耗 · 结束后自动关 | 用户说"开"就开 |
| Token 预算 | Agent 建议预算 · 50/70/90%预警 · 降级链建议 | 用户可提高或取消限制 |
| Pi 安全 | Balanced 默认 · 危险拦截不可绕过 | 用户可切换到 Permissive |
| 模型选择 | 力点探测推荐 · 降级链 | 用户手动选任何模型 |
| Skill 管理 | 按需唤醒/休眠 · 匹配度排序 · 低效自动降权 | 装不装、卸不卸 |

**"一"的价值来自"四十九"**：平台不是替用户做决定，是帮用户把决定做得更聪明。

### 0.9 Skill 三层自生态

| 层次 | 内容 | 状态 |
|:--|:--|:--:|
| 吸纳层 | 社区 Skill → 力点探测 → 按需唤醒/休眠 | 接口已验证 ✅ |
| 自开发 | 执行经验 → 自动提取模式 → 生成新 SKILL.md | 📋 |
| 自进化 | 使用中持续优化 → 优胜劣汰 → 越用越好用 | 📋 v2.0 |

核心原则：**安装≠启动**。Skill 不自动全开——每个任务只唤醒最匹配的少数几个，其余休眠。任务结束立即释放上下文。

### 0.10 与竞品的本质区别

> 每个竞品都在做"更强"的 Agent。超级牛马做的是"更聪明"的 Agent——不烧 Token、不打扰用户、不越界、但越用越强。太极不是外壳，是引擎。

## 3. 工作间沙盒系统

### 3.1 沙盒定义

每个工作间是一个**独立执行环境**，具备完整的资源隔离。

| 资源 | 隔离方式 | 状态 |
|:--|:--|:--:|
| 任务队列 | 工作间内独立队列，WS-1 的任务不影响 WS-2 | ✅ |
| 记忆 | 工作间作用域，Sub-Agent 记忆仅在本工作间有效 | ✅ |
| 文件 | 工作间目录隔离，每工作间独立文件空间 | ✅ |
| 对话上下文 | 独立上下文，不可访问其他工作间历史 | ✅ |
| Agent 通信 | 同工作间内可直接通信，跨工作间必须经 Hermes 中转 | ✅ |
| 模型配置 | 每工作间可独立指定默认模型 | ✅ |

### 3.2 工作间生命周期

```
创建（用户/"+"菜单）
  │
  ▼
配置（名称/描述/默认模型/Agent 列表）
  │
  ▼
运行（接收任务，Agent 执行）
  │
  ▼
归档（保留历史，停止新任务）
  │
  ▼
删除（清理所有资源）
```

### 3.3 群聊（跨工作间协作）

| 属性 | 值 |
|:--|:--|
| 定位 | 跨工作间 Agent 协作的唯一通道 |
| 通信方式 | Hermes 中转，不可绕过 |
| 参与者 | 来自不同工作间的 Agent |
| 状态 | ✅ 已实现（UI 已完成） |

## 7-A. Token 消耗管控铁律

> **这是超级牛马对 Hermes Agent 和所有 Sub-Agent 的 Token 管控层。
> Hermes 原生自带 Token 优化（双层压缩、提示词缓存等），但超级牛马的平台级管控在其之上。**

### 7A.1 管控总则

```
┌─────────────────────────────────────────────┐
│  超级牛马平台级 Token 管控                     │
│  ┌───────────────────────────────────────┐  │
│  │ 能力开关（Capability Flags）           │  │
│  │ web_fetch → 默认关闭，按需开启          │  │
│  │ web_search → 默认关闭，按需开启         │  │
│  │ mcp → 按 Agent 粒度审批                │  │
│  │ subagents → 有预算上限                  │  │
│  └───────────────────────────────────────┘  │
│  ┌───────────────────────────────────────┐  │
│  │ Hermes 原生 Token 优化（继承）          │  │
│  │ 双层压缩(85%+50%) / 提示词缓存 / undo  │  │
│  │ /undo / FTS5段合并 / 紧凑行号          │  │
│  └───────────────────────────────────────┘  │
│  ┌───────────────────────────────────────┐  │
│  │ 超级牛马叠加层                          │  │
│  │ Agent维度的Token预算 / 降级链 / 告警  │  │
│  │ 外网访问封锁 / 国产模型优先             │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

### 7A.2 能力开关（Capability Flags）— 外网控制核心

Hermes v0.16.0 支持通过 `capabilities` 开关控制 Agent 能力。超级牛马在此基础上建立**三级能力开关**：

| 能力 | 全局默认 | Agent 可申请 | 说明 |
|:--|:--:|:--:|:--|
| `web_fetch` | **❌ 关闭** | ✅ 可申请 | 访问外网抓取网页——如无必要，绝不开启 |
| `web_search` | **❌ 关闭** | ✅ 可申请 | 搜索引擎查询——默认关闭，国产搜索引擎优先 |
| `mcp` | **❌ 关闭** | ✅ 按 Agent 审批 | MCP 外部工具接入——每 Agent 独立审批 |
| `skills` | ✅ 开启 | ✅ | 技能调用——默认开放，使用 Hermes 技能生态 |
| `attachments` | ✅ 开启 | ✅ | 图片附件——默认开放 |
| `memory` | ✅ 开启 | ✅ | 跨会话记忆——默认开放 |
| `subagents` | 📋 按预算 | ✅ 按预算 | 子 Agent 委派——有 Token 预算上限 |

**外网访问铁律**：

> 1. **web_fetch 和 web_search 全局默认关闭。**
> 2. Agent 执行任务时，如无必要，绝不访问外网。
> 3. 若任务确实需要外网数据，Agent 必须先向用户说明原因，用户审批后才开启。
> 4. 外网访问结果必须缓存，同一 URL 24h 内不重复请求。
> 5. 优先使用本地知识库和记忆系统回答问题，避免外网查询。

### 7A.3 Token 预算体系

#### Agent 维度 Token 预算

| Agent 类型 | 单次任务预算 | 日预算 | 说明 |
|:--|:--:|:--:|:--|
| Hermes Agent（平台助手） | 50,000 | 200,000 | Root Node，编排任务消耗 |
| Writer Sub-Agent | 30,000 | 100,000 | 写作类任务 |
| Coder Sub-Agent | 50,000 | 150,000 | 编码类任务 |
| Analyst Sub-Agent | 40,000 | 120,000 | 分析类任务 |
| 其他 Sub-Agent | 20,000 | 80,000 | 轻量任务 |

#### 预算消耗提醒

| 消耗比例 | 行为 |
|:--:|:--|
| 0-50% | 正常执行，无提醒 |
| 50-70% | ⚠️ 上下文压力警告（Hermes 原生） |
| 70-90% | 🔶 预算预警，通知用户 |
| 90-100% | 🔴 任务暂停，等待用户确认是否继续 |
| 100%+ | ❌ 自动降级到更省 Token 的模型或停止 |

### 7A.4 上下文压缩叠加策略

超级牛马在 Hermes 原生的双层压缩之上做三层叠加：

```
Layer 3（超级牛马叠加）: Agent 维度 Token 预算 + 自动降级
Layer 2（Hermes 原生）: Agent ContextCompressor（50% 阈值）
Layer 1（Hermes 原生）: Gateway 会话清理（85% 阈值）
```

| 叠加策略 | 说明 | 状态 |
|:--|:--|:--:|
| 对话摘要注入 | Hermes 四阶段压缩的结构化摘要同步到 L2 记忆 | ✅ |
| 冗余工具输出清理 | > 200 字符的旧工具结果替换为占位符（Hermes 原生） | ✅ |
| 国产模型优先 | 国产模型 Token 成本通常为 Claude/GPT 的 1/5~1/10 | ✅ |
| 提示词固化 | 稳定的系统 prompt 和工具 schema 被缓存（Hermes 原生 cache-control） | 📋 |
| 工具调用去重 | 相同参数的工具调用 5 分钟内不重复执行 | 📋 v2.0 |
| 模型降级链 | Token 超限时自动切换到更便宜的模型 | 📋 v2.0 |

### 7A.5 Token 监控与可视化

| 功能 | 说明 | 状态 |
|:--|:--|:--:|
| 实时 Token 消耗面板 | 每个 Agent 的当前/累计 Token | ✅ 已实现 |
| 日/周/月 Token 报告 | 按 Agent/工作间/模型分类统计 | ✅ 已实现 |
| 成本估算 | 根据模型定价实时换算人民币 | 📋 v2.0 |
| 异常消耗告警 | Token 消耗增速异常时通知 | 📋 v2.0 |
| 节省建议 | 根据使用模式推荐优化方案 | 🔮 v2.0+ |

### 7A.6 模型降级链（Token不足时自动切换）

```
用户指定模型（优先级最高）
  │
  ▼
Token 消耗 < 50% 预算？
  ├── 是 → 继续使用当前模型
  │
  └── 否 → 降到下一级
         │
         ▼
工作间默认模型
```

## 8. Swarm 编排引擎

### 8.1 编排拓扑

```
用户指令
  │
  ▼
┌──────────────────────┐
│  Root Node            │  Hermes Agent
│  意图解析 + 任务分解   │  将用户指令拆解为子任务树
└──────┬───────────────┘
       │
  ┌────┼────┐
  ▼    ▼    ▼
┌────┐┌────┐┌────┐
│ W1 ││ W2 ││ W3 │     Worker（Sub-Agent）
│    ││    ││    │     并行执行子任务
└──┬─┘└──┬─┘└──┬─┘
   │     │     │
   ▼     ▼     ▼
┌──────────────────────┐
│  Gate Validator       │  质量门禁
│  验证 Worker 输出质量  │  PASS / FAIL / REWORK
└──────┬───────────────┘
       │ PASS
       ▼
┌──────────────────────┐
│  Gate Synthesizer     │  结果合成
│  合成子结果为最终交付物 │
└──────┬───────────────┘
       │
       ▼
┌──────────────────────┐
│  Shared Blackboard    │  共享上下文
│  工作间级共享状态      │  Agent 间通信中介
└──────────────────────┘
```

### 8.2 质量门禁决策

```
Worker 产出
  │
  ▼
Gate Validator 校验
  │
  ├── PASS → Synthesizer 合成 → 交付用户
  │
  ├── FAIL → 记录原因 → 通知用户 → 停止
  │
  └── REWORK → 自动重试（最多 3 次）
                │
                ├── 3 次内 PASS → 正常交付
                │
                └── 3 次仍失败 → PASS_WITH_WARNING + 标记
```

|:--|:--|:--:|
| Root Node | Hermes Agent | ✅ |
| Worker | 工作间 Sub-Agent | ✅ |
| Gate Validator | 质量门禁 | ✅ |
| Gate Synthesizer | 编排器 | ✅ |

## 14. 数据流设计

### 14.0 Smart Allocator 接口（v1.6 落地 — 实验已验证 ✅）

> 力点探测（Force Detection）是 Smart Allocator 的核心算法。实验二 PoC 已跑通 3/3 核心断言。详见 `experiments/force-detection-poc.py`。

#### 接口定义

```python
@dataclass
class ForceProbeInput:
    task_type: TaskType          # 任务类型
    budget_remaining: int        # 当前日预算剩余
    user_priority: float         # 用户优先级（0-1）

@dataclass  
class ForceProbeResult:
    budget_level: BudgetLevel    # HIGH / STANDARD / LOW
    recommended_model: str       # 推荐模型名称
    estimated_tokens: int        # 预分配 Token 数
    reason: str                  # 决策理由
    confidence: float            # 置信度（0-1）
```

#### 决策算法

```
综合得分 = 用户优先级 × 0.4 + ROI × 0.35 + 预算比例 × 0.25

边界规则：
  低优先级 + 预算紧张 → LOW（Gemma-4）
  低价值对话 + 优先级 < 0.6 → LOW（Gemma-4）

高得分路径：
  score > 0.45 或 (优先级 > 0.7 且 历史质量 > 0.7) → HIGH
  score > 0.25 → STANDARD
  否则 → LOW
```

#### 验证结果

```
核心编码（高优先+预算充裕）→ HIGH · DeepSeek V3.2 · 50K tokens  ✅
低优先搜索（预算紧张）     → LOW  · Gemma-4     · 3K tokens   ✅
日常闲聊                   → LOW  · Gemma-4     · 2K tokens   ✅
```

```

  │
  ▼
Worker Sub-Agent 执行（并行）
  │
  ▼
Gate Validator 校验
  │
  ▼
Gate Synthesizer 合成
  │
  ▼
SSE 流式输出 → 前端渲染
```

```
  │
```
