# 太极引擎（Taiji Engine）——超级牛马工作台底层核心引擎

> **定位**：太极引擎是超级牛马工作台的底层运转核心。七律驱动的自平衡、自优化、自进化系统。
> **日期**：2026-06-12 | **状态**：架构设计，待实现

---

## 一、太极引擎是什么

太极引擎不是"太极风格的 UI"，也不是"太极隐喻的命名"。它是超级牛马工作台的**底层运转机制**——用太极原理驱动系统的自平衡、自优化、自进化。

```
太极引擎 = 七律驱动的自动进化系统

    阴阳平衡 → 动态均衡器
    四两拨千斤 → 智能资源调度器
    以静制动 → Hermes 注意力机制
    无为而治 → 自愈回路
    顺势而为 → 借力架构
    刚柔并济 → 治理层
    生生不息 → 进化循环
```

## 二、七律下沉到架构层

### 2.1 阴阳平衡 → 动态均衡器（Dynamic Balancer）

**哲学**：阴阳不是固定的，是动态转化的。系统永远在两个极端之间寻找最优平衡点。

**架构实现**：

```
┌──────────────────────────────────────────────────┐
│           Dynamic Balancer v2.0                   │
│                                                   │
│   ┌──────────┐     ┌──────────┐    ┌──────────┐  │
│   │ 阴（本地）│ ←─→ │ 和（网格）│ ←─→ │ 阳（云端）│  │
│   │ Ollama   │     │ Taiji Mesh│    │ API      │  │
│   │ 隐私优先 │     │ 算力共享 │    │ 算力最强 │  │
│   └──────────┘     └──────────┘    └──────────┘  │
│                                                   │
│   三层路由：本地 → 太极网格 → 云端                   │
│   均衡因子：任务复杂度 × 隐私敏感度 ×                  │
│             Token 预算剩余 × 网格节点可用性            │
└──────────────────────────────────────────────────┘
```

**运行规则**：

| 均衡维度 | 阴极端 | 阳极端 | 均衡算法 |
|:--|:--|:--|:--|
| 模型选择 | 本地 Ollama | 云端 API | `权重 = 隐私敏感度 × 0.6 + 任务复杂度 × 0.3 + 预算剩余 × 0.1` |
| Hermes 介入 | 沉默观察 | 主动干预 | `介入概率 = 异常程度 × 置信度倒数`——越确定的事越沉默 |
| 工作间隔离 | 硬隔离 | Hermes 穿透 | 默认隔离，穿透需要"必要性评分"≥阈值 |

**为什么是太极**：不选边站。本地或云端、沉默或介入——不是二选一，是动态平衡。太极图的黑白不是"一半一半"，是"当前这一刻，黑多一点还是白多一点，由当下状态决定"。

### 2.2 四两拨千斤 → 智能资源调度器（Smart Allocator）

**哲学**：不是省资源，是找到力的方向。最少的投入，最大的产出。

**架构实现**：

```
任务进入
  │
  ▼
力点探测（Force Detection）
  │
  ├── 任务类型识别（写作/编码/分析/搜索？）
  ├── 历史相似任务 Token 消耗分析
  ├── 当前可用模型性价比排序
  └── 任务产出预期评估
  │
  ▼
资源分配决策
  │
  ├── 低ROI任务 → Gemma-4 + 最小 Token 预算
  ├── 中ROI任务 → DeepSeek V3.2 + 标准预算
  ├── 高ROI任务 → 最高可用模型 + 弹性预算
  └── 外网搜索 → 必须审批 + 24h 缓存优先
  │
  ▼
执行中监控
  │
  ├── 偏离预期？→ 提前终止或降级
  ├── 产出高质量？→ 提升预算
  └── 卡死循环？→ 强制中断
```

**力点探测的维度**：

| 维度 | 探测方法 | 低ROI特征 | 高ROI特征 |
|:--|:--|:--|:--|
| 任务复杂度 | 意图解析 | 简单问答 | 跨域编排 |
| 历史相似度 | L2 记忆检索 | 已有缓存答案 | 全新问题 |
| 产出预期 | 任务目标清晰度 | 探索性任务 | 确定性产出 |
| 用户偏好 | Honcho 建模 | 用户不关注 | 用户核心需求 |

**为什么是太极**：太极推手的精髓是"听劲"——不是先出招，是先感知力的方向，然后顺势发力。Smart Allocator 同样——不是先分预算，是先"听"这个任务值多少 Token，然后精准分配。

### 2.3 以静制动 → Hermes 注意力机制（Attention Engine）

**哲学**：不动如山，动如雷震。不是不说话，是在对的时刻说对的话。

**架构实现**：

```

│       Hermes Attention Engine             │
│                                           │
│  ┌─────────────────────────────────────┐  │

│  │  监控所有工作间的：                    │  │
│  │  · Agent 状态                        │  │
│  │  · Token 消耗                        │  │
│  │  · 质量门禁结果                      │  │
│  │  · 异常事件                          │  │
│  │  · 用户在线状态                      │  │
│  └──────────────┬──────────────────────┘  │
│                 │                         │
│                 ▼                         │
│  ┌─────────────────────────────────────┐  │
│  │       注意力阈值判断                  │  │
│  │  只有在以下情况触发主动说话：           │  │
│  │                                      │  │
│  │  1. 破冰：用户首次/每日首次打开        │  │
│  │  2. 进度：长任务需要阶段性反馈          │  │
│  │  3. 预警：Token/安全阈值被突破         │  │

│  │  6. 发现：跨工作间共性模式识别         │  │
│  │  7. 建议：根据用户行为提出优化         │  │
│  └──────────────────────────────────────┘  │
└───────────────────────────────────────────┘
```

**为什么是太极**：太极讲"后发先至"——不是不做反应，是等对方先出手，看清了再出手，出手就中。Hermes Attention Engine 同理——它知道一切，但只在时机成熟时开口。不是话痨，是狙击手。

### 2.4 无为而治 → 自愈回路（Self-Healing Loop）

**哲学**：最好的治理是不治理。让系统自己发现问题、纠正问题、预防问题。

**架构实现**：

```
正常执行
  │
  ▼
Pi 准则监控（始终运行）
  │
  ├── 常规操作 → 自动放行
  ├── 灰色地带 → 评估风险分
  └── 危险操作 → 拦截
        │
        ▼
   ┌────────────────────────┐
   │    自愈回路             │
   │                        │
   │  拦截不是终点，          │
   │  是学习的起点：          │
   │                        │
   │  1. 记录拦截原因        │
   │  2. 分析 Agent 意图     │
   │  3. 生成替代方案        │
   │  4. 提交给 Agent        │
   │  5. Agent 用替代方案     │
   │     继续执行            │
   │  6. 沉淀为新的安全规则   │
   │                        │
   └────────────────────────┘
```

**自愈回路的运行效果**：

| 场景 | 传统方式 | 自愈回路 |
|:--|:--|:--|
| Agent 越界访问文件 | 弹窗→用户审批→放行/拒绝 | 分析意图→生成替代方案→Agent自纠→规则沉淀 |
| Token 超预算 | 硬中断 | 自动降级模型→继续执行→事后报告 |
| 编排死锁 | 超时失败 | 检测循环→调整拓扑→继续 |
| 重复工具调用 | 浪费 Token | 检测重复→缓存结果→自动跳过 |

**为什么是太极**：老子说"太上，下知有之"——最好的统治者，人民只知道他的存在。自愈回路就是"用户不知道 Hermes 做了什么，但系统就是不出问题"。

```
  │
  └── 前端 UI 层 ────────── 借力 Hermes v0.16.0 Desktop
       └── Electron 跨平台 / 中文界面 / Web Dashboard
```

**借力的收益计算**：

| 借力来源 | 节省的工作量 | 获得的品质 |
|:--|:--|:--|
| Hermes Agent | 省去自研 Agent 运行时（估计 5000+ 人时） | 542 PR 实战检验的稳定性 |

| Hermes Desktop | 省去桌面壳层开发（估计 1500+ 人时） | 跨平台 + 中文 + 自更新 |

**为什么是太极**：不是"我们比别人强"，是"我们知道力在哪"。顺势而为不是偷懒，是智慧——把有限的力气花在独有的护城河上（Token管控、记忆闭环、Swarm编排），通用的部分交给生态系统。

### 2.6 刚柔并济 → 双轨治理层（Dual-Track Governance）

**哲学**：硬的该硬，软的该软。刚性约束和柔性引导并行。

**架构实现**：

```
┌──────────────────────────────────────────┐
│         Dual-Track Governance            │
│                                          │
│  刚性轨道（Hard Track）                   │
│  ┌────────────────────────────────────┐  │
│  │ Capability Flags（web_fetch ❌）    │  │
│  │ Token 硬预算上限                    │  │
│  │ Pi 准则三关拦截                    │  │
│  │ 工作间沙盒硬隔离                    │  │
│  │ 操作审计日志（不可篡改）             │  │
│  └────────────────────────────────────┘  │
│                                          │
│  柔性轨道（Soft Track）                   │
│  ┌────────────────────────────────────┐  │
│  │ 记忆积累（越用越懂）                 │  │
│  │ 模型降级链（软切换）                 │  │
│  │ Honcho 偏好学习                     │  │
│  │ 技能自优化（使用中改进）              │  │
│  │ Hermes 穿透协调（软协调）             │  │
│  └────────────────────────────────────┘  │
│                                          │
│  交互规则：                               │
│  · 刚柔不互斥——同一条操作可以硬拦截       │
│    但软记录原因                          │
│  · 柔性积累到阈值 → 触发刚性优化           │
│  · 刚性拦截发现模式 → 转化为柔性偏好       │
└──────────────────────────────────────────┘
```

**为什么是太极**：纯刚则脆，纯柔则散。太极讲"刚柔相济"——最好的结构是"外柔内刚"（用户体验柔软，底层约束刚硬）或"外刚内柔"（安全边界刚硬，内部演化柔软）。

### 2.7 生生不息 → 进化循环（Evolution Loop）

**哲学**：太极图是旋转的。系统不是一个固定态，是一个永动的自进化循环。

**架构实现**：

```
                ┌──────────────┐
                │   执行        │
                │ Agent 完成任务│
                └──────┬───────┘
                       │
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
    ┌─────────┐  ┌─────────┐  ┌─────────┐
    │ 沉淀     │  │ 反思     │  │ 传导     │
    │ L2/L3   │  │ 模式识别 │  │ 更新技能 │
    └────┬────┘  └────┬────┘  └────┬────┘
         │            │            │
         └────────────┼────────────┘
                      │
                      ▼
                ┌──────────────┐
                │   进化        │
                │ 下一次执行    │
                │ 能力和效率    │
                │ 都更好了      │
                └──────────────┘
```

**每一圈的进化量**：

| 进化维度 | 沉淀什么 | 效果 |
|:--|:--|:--|
| **记忆进化** | 成功路径、失败原因、关键决策 | 下次同类任务不走弯路 |
| **技能进化** | 高频操作模式、最优参数 | 技能自动调优，越用越准 |
| **编排进化** | Worker 组合效果、门禁阈值 | 编排策略越来越高效 |
| **模型进化** | 每模型的 ROI 跟踪 | 自动选择性价比最高的模型 |
| **安全进化** | 拦截模式→新的 Pi 规则 | 安全边界自适应 |
| **用户进化** | Honcho 偏好建模 | 越用越懂用户 |

**为什么是太极**：太极图在旋转——黑转到白，白转到黑，永远不停止。进化循环就是超级牛马的旋转——每一次任务执行，都让系统比上一次更聪明。

### 2.8 算力三元 → 太极网格（Taiji Mesh）🆕 2026-06-24

**哲学**：算力不是"我有或没有"的二元问题，是"如何从网络中生出第三种力量"的三元结构。

**架构实现**：

```
┌──────────────────────────────────────────────────┐
│              Taiji Mesh（太极网格）                │
│                                                   │
│   三层路由：本地 → 网格 → 云端                      │
│                                                   │
│   ┌──────────┐     ┌──────────┐    ┌──────────┐  │
│   │ 第一层    │ ──→ │ 第二层    │ ──→ │ 第三层    │  │
│   │ 本地推理  │     │ P2P 网格 │     │ 云端 API │  │
│   │ 0成本     │     │ 低成本    │     │ 保底     │  │
│   │ 0延迟     │     │ 低延迟    │     │ 能力强   │  │
│   └──────────┘     └──────────┘    └──────────┘  │
│                                                   │
│   节点生命周期：探测→注册→心跳→服务→摘除              │
│   贡献计量：GPU秒 × 系数 = 太极积分                  │
│   隐私保证：端到端加密 + 沙盒执行 + 退出即消失        │

```

**三层路由决策**：

| 路由层 | 条件 | 成本 | 延迟 | 适用场景 |
|:--|:--|:--|:--|:--|
| 本地 | 有匹配模型 | 0 | <10ms | 隐私敏感、高频任务 |
| 网格 | 邻居节点有模型 | 积分 | <50ms | 本地无模型、预算紧张 |
| 云端 | 网格不可用 | API费 | <200ms | 复杂任务、保底路径 |

## 六、一句话总结

> 太极引擎不是"用太极包装产品"，而是**用太极原理设计产品的运转机制**。
>
> 每个竞品都在做"更强"的 Agent。
> 超级牛马做的是"更聪明"的 Agent——不烧 Token、不打扰用户、不越界、但越用越强。
>
> 太极不是外壳，是引擎。
